# CLAUDE.md

Referensi arsitektur & konvensi untuk **Speedora** — platform AI video repurposing (mirip OpusClip) yang mengubah video panjang menjadi klip pendek siap-viral secara otomatis.

## Ringkasan Produk

Alur inti MVP:

```
Upload video -> Transcript (ASR) -> Auto-clip (deteksi momen menarik) -> Caption (burn-in) -> Download
```

## Tech Stack

| Layer | Teknologi |
|---|---|
| Frontend | Next.js + TypeScript |
| Backend API | NestJS |
| Database | PostgreSQL (via Prisma ORM di `packages/database`, dipakai `apps/api` & `apps/worker`) |
| Queue / Cache | Redis + BullMQ |
| Video processing | FFmpeg cluster (worker nodes terpisah) |
| ASR (speech-to-text) | Whisper — Groq `whisper-large-v3-turbo` (default, gratis) atau OpenAI `whisper-1` (premium, berbayar) |

## Struktur Monorepo

```
apps/
  web/        # Next.js frontend — upload UI, editor klip, preview, dashboard
  api/        # NestJS backend — REST/GraphQL API, auth, job orchestration
  worker/     # Job consumer — ASR (Whisper), auto-clip detection, FFmpeg render, captioning
packages/
  shared/     # Tipe TypeScript, DTO, konstanta, util yang dipakai lintas apps
  database/   # Prisma schema/client Postgres, dipakai apps/api dan apps/worker
  storage/    # Klien object storage S3-compatible (upload/download/delete), dipakai apps/api dan apps/worker
  social/     # OAuth client, enkripsi token, upload klien per-platform (YouTube, TikTok, Instagram), dipakai apps/api dan apps/worker
```

- `apps/web` dan `apps/api` hanya berkomunikasi lewat HTTP API — tidak ada import langsung antar keduanya.
- `apps/worker` tidak melayani HTTP; hanya mengonsumsi job dari BullMQ queue yang di-enqueue oleh `apps/api`.
- Tipe yang dibagi antara frontend, backend, dan worker (mis. bentuk payload job, status enum, DTO) didefinisikan sekali di `packages/shared` — jangan duplikasi tipe di masing-masing app.

## Alur Pemrosesan Video (MVP)

1. **Upload** — `apps/web` upload file ke `apps/api`, video disimpan (object storage), record dibuat di PostgreSQL dengan status `UPLOADED`.
2. **Transcript** — `apps/api` enqueue job `transcribe` ke BullMQ. `apps/worker` menjalankan Whisper, hasil transcript (dengan timestamp) disimpan ke PostgreSQL. Status -> `TRANSCRIBED`.
3. **Auto-clip** — job `detect-clips` (dienqueue oleh `apps/worker` sendiri begitu `transcribe` sukses, bukan oleh `apps/api`) mengirim transcript ke LLM (GPT) untuk memilih 1-3 momen paling menarik/viral-worthy sebagai kandidat klip (start/end timestamp + virality score 0-100 + `hookText`/`hashtags`, lihat "Hook & Hashtag Generator (Fase 5)" di bawah). Menghasilkan daftar kandidat klip. Status -> `CLIPS_DETECTED`.
4. **Caption + Reframe** — untuk tiap kandidat klip, `detect-clips` yang sukses langsung enqueue satu job `render-clip` (self-chain, sama seperti `transcribe` -> `detect-clips`). `render-clip` mendeteksi wajah di sepanjang rentang klip (lihat "Smart Reframe (Fase 2)" di bawah), memotong+crop 9:16 video dengan FFmpeg sesuai timestamp klip dan posisi wajah, generate file ASS dari transcript sesuai `Clip.captionStyle` (lihat "Caption Styling (Fase 3)" di bawah; timestamp digeser relatif ke awal klip), lalu burn-in sebagai subtitle ke video hasil crop. `Clip.outputUrl` di-set setelah render sukses; `Video.status` -> `RENDERED` baru setelah **semua** klip milik video tersebut selesai di-render (bukan begitu klip pertama selesai).
5. **Download** — `apps/web` polling `GET /videos/:id` tiap 2 detik sampai status `RENDERED` atau `FAILED`, lalu menampilkan daftar klip dengan link `GET /clips/:id/download` (`apps/api` streaming file dari object storage, key-nya ada di `Clip.outputUrl`).
6. **Retry** — kalau status `FAILED`, `apps/web` menampilkan tombol Retry yang manggil `POST /videos/:id/retry`. Tahap yang di-retry disimpulkan dari data yang sudah ada (bukan dari marker "gagal di tahap mana" yang disimpan terpisah): belum ada `TranscriptSegment` -> retry `transcribe`; ada segment tapi belum ada `Clip` -> retry `detect-clips`; ada `Clip` tapi sebagian belum punya `outputUrl` -> retry `render-clip` **hanya** untuk klip yang belum itu (tiap klip render independen, jadi satu klip gagal tidak berarti klip lain ikut di-retry). Aman disimpulkan begitu karena `transcribe`/`detect-clips` masing-masing nulis datanya sekaligus naikkan status di step yang sama — kalau job-nya gagal (masuk `catch`), data tahap itu belum sempat ketulis sama sekali. Lihat `VideosService.retry`.

Setiap tahap adalah job terpisah di BullMQ (bukan satu job monolitik) agar retry granular per-tahap dan agar FFmpeg cluster bisa discale independen dari proses ASR.

## Timeline Editor (Fase 1 pasca-MVP)

Route `/videos/:id/edit` di `apps/web` — fine-tune hasil auto-clip sebelum download, tanpa perlu re-upload atau menunggu pipeline ulang dari awal.

- **Preview**: `<video>` (stream source asli via `GET /videos/:id/source`, bukan hasil render) + `<canvas>` overlay di atasnya untuk caption, di-redraw tiap `requestAnimationFrame` mengikuti `video.currentTime`. **Approximate, bukan pixel-perfect** terhadap hasil burn-in libass FFmpeg (font/posisi/outline sederhana) — tetap approximate setelah Fase 3 (custom caption styling) juga: preview canvas belum ikut menampilkan efek karaoke/bold-highlight per-kata, cuma teks polos bold putih apa pun `captionStyle`-nya. Membangun shared style engine antara canvas dan ASS override FFmpeg dianggap scope creep untuk fase ini — lihat "Caption Styling (Fase 3)" di bawah.
- **Timeline**: track klip (bar per klip hasil `detect-clips`, drag handle start/end untuk klip yang sedang dipilih) + track caption (transcript segment yang overlap klip terpilih, visual saja, tidak draggable). Dibangun dengan div ber-posisi absolut + persentase (bukan SVG/canvas) untuk timeline-nya sendiri — cuma overlay caption di atas video yang pakai canvas.
- **State**: Zustand (`apps/web/lib/timelineStore.ts`) — cukup ringan untuk state timeline (klip + draft trim + flag dirty/saving/rendering per klip) tanpa perlu Redux penuh.
- **Trim manual TIDAK auto re-render**: `PATCH /clips/:id` (`ClipsService.update`) cuma update `startTime`/`endTime` di DB. Render ulang adalah aksi eksplisit terpisah (`POST /clips/:id/render`) supaya geser slider tidak buang compute FFmpeg tiap perubahan kecil.
- **Re-render reuse job `render-clip` yang sudah ada**, bukan job baru — `render-clip.worker.ts` sudah generik atas start/end dan overwrite key `renders/<clipId>.mp4` yang sama, jadi tidak ada bedanya secara teknis antara render pertama (dari `detect-clips`) dan re-render manual (dari editor). `ClipsService.render` men-clear `Clip.outputUrl` ke `null` **sebelum** enqueue — bukan cuma dibiarkan stale — supaya dua hal yang sudah ada otomatis benar tanpa ubah kode lain: tampilan "Rendering..." di dashboard (sudah ada untuk `downloadUrl: null`) langsung kepakai, dan kalau render-clip job ini gagal, `VideosService.retry`'s pengecekan "klip tanpa `outputUrl` perlu di-render lagi" tetap akurat (bukannya mengira klip ini masih fine karena `outputUrl` lama belum dihapus).
- **`GET /videos/:id/source`** streaming source video (bukan hasil render) dengan dukungan HTTP Range (`packages/storage`'s `getObjectStreamRange`) supaya `<video>` bisa scrub/seek tanpa download seluruh file dulu — `<video crossOrigin="use-credentials">` di frontend supaya cookie sesi ikut terkirim cross-origin (api beda port dari web). Proxy lewat `apps/api` (bukan presigned URL langsung ke R2), konsisten dengan keputusan storage yang sudah ada: client tidak pernah akses bucket langsung.
- **`GET /videos/:id/transcript`** endpoint terpisah dari `findOne()`/`GET /videos/:id` — sengaja tidak digabung supaya endpoint yang di-polling tiap 2 detik (upload progress, dashboard) tidak ikut membawa payload transcript (bisa banyak baris untuk video panjang) padahal cuma editor yang butuh teksnya.
- **`filterSegmentsForClip`** (overlap-filter transcript-untuk-rentang-klip) diekstrak ke `packages/shared/src/utils/transcript.ts` — sebelumnya duplikat identik di `detect-clips.worker.ts` dan `VideosService.retry`; ClipsService.render jadi pemakai ketiga, titik di mana duplikasi lebih dari cukup untuk diekstrak jadi util bersama.

## Smart Reframe (Fase 2 pasca-MVP)

Setiap `render-clip` job sekarang men-crop video ke 9:16 mengikuti wajah yang paling menonjol di frame, bukan sekadar center-crop statis dari aspect ratio asli.

- **Inline di job `render-clip` yang sudah ada, bukan job BullMQ baru** (`detect-faces`) — deteksi wajah adalah sub-step di awal `render-clip` (posisinya setara dengan `buildSrt()` yang sudah ada: dihitung, dipakai, lalu buang, tidak disimpan lintas job). Tidak perlu schema baru di `Clip`, tidak perlu cabang baru di `VideosService.retry` — kalau `render-clip` gagal (termasuk karena deteksi wajah), seluruh job itu yang di-retry sebagai satu unit, sama seperti perilakunya sebelum fase ini.
- **MediaPipe dijalankan lewat subprocess Python asli** (`apps/worker/scripts/detect_faces.py`, dipanggil dari `apps/worker/src/faceDetection.ts` via `child_process` — pola yang sama persis dengan `apps/worker/src/ffmpeg.ts` manggil binary `ffmpeg`), bukan lewat `@mediapipe/tasks-vision` (WASM/Node). MediaPipe secara resmi paling matang di Python/browser, bukan Node server-side. Pakai **Tasks API** (`mediapipe.tasks.python.vision.FaceDetector`) — API lama `mp.solutions.face_detection` sudah **dihapus** di rilis mediapipe terbaru (0.10.x). Tasks API butuh file model `.tflite` terpisah (tidak dibundel di pip package) — di-download dari model zoo MediaPipe, disimpan di `apps/worker/models/` (gitignored, bukan aset yang di-commit; lihat README.md untuk cara download-nya).
- **`node:20-alpine` -> `node:20-slim` (Debian/glibc) untuk image `apps/worker`** — MediaPipe's PyPI wheels **tidak ada build musl (Alpine) sama sekali** (`pip install mediapipe` di Alpine langsung gagal "No matching distribution"). `opencv-python-headless` juga butuh beberapa shared library sistem meski headless (`libxcb1 libxext6 libsm6 libgl1 libglib2.0-0 libgomp1 libxrender1 libegl1 libgles2` — ditemukan lewat trial-and-error terhadap container asli, bukan disalin dari satu sumber tunggal) dan `pip install --break-system-packages` (Debian 12 menerapkan PEP 668 untuk system Python).
- **Deteksi wajah tidak boleh menggagalkan job** — `detectFaces()` di-try/catch dalam `buildReframePlan()` (`render-clip.worker.ts`); kalau subprocess Python gagal karena alasan APAPUN (bukan cuma "tidak ada wajah", tapi juga mis. environment bermasalah), fallback ke center-crop statis, bukan melempar error. Ini perluasan alami dari requirement "jangan gagal cuma karena tidak ada wajah" ke "jangan gagal karena mekanisme deteksinya sendiri bermasalah".
- **Heuristik "wajah paling menonjol", bukan deteksi speaker aktif sungguhan** — kalau ada lebih dari satu wajah di satu frame sampel, dipilih yang bounding box-nya paling besar (proxy untuk "yang bicara"). Deteksi speaker aktif sungguhan (perlu Face Landmarker + korelasi audio) di luar scope fase ini.
- **Sampling ~1 frame/detik** (`FACE_SAMPLE_INTERVAL_SECONDS` di `apps/worker/src/faceDetection.ts`) — klip auto-detect sudah dibatasi 5-60 detik oleh prompt LLM `detect-clips`, jadi maksimal ~60 titik sampel per klip. Python baca frame langsung lewat OpenCV `VideoCapture.set(CAP_PROP_POS_MSEC, ...)` (seek langsung, bukan ekstrak semua frame ke file gambar dulu) — jadi `reserveScratchPath` yang sudah ada **tetap cukup dipakai apa adanya** tanpa perlu direktori scratch baru untuk frame; satu-satunya scratch file baru adalah command file `sendcmd` (lihat di bawah).
- **Interpolasi crop-path dihitung di TypeScript** (`apps/worker/src/reframe.ts`, step lebih rapat — `CROP_PATH_STEP_SECONDS` 0.2 detik — daripada sampling deteksi wajah), bukan sebagai satu ffmpeg expression yang panjang. Hasilnya di-generate jadi file command `sendcmd` (satu baris per titik interpolasi: `TIME crop@reframe x X, crop@reframe y Y;`) yang dipasang sebelum filter `crop` yang di-tag `@reframe` di filtergraph FFmpeg (`crop@reframe=w=W:h=H:x=X0:y=Y0`) — diverifikasi langsung dengan video test pattern sungguhan (posisi crop terbukti benar-benar bergerak, bukan cuma diam di X0/Y0).
- **Fallback ke center-crop statis** (tanpa `sendcmd`, filter `crop=` polos) kalau tidak ada wajah terdeteksi di **satu pun** sampel sepanjang klip (`buildCropPath()` return `null`) — bukan cuma error handling, tapi juga menghindari generate crop-path yang sebenarnya konstan (percuma).
- **Crop cuma gerak di satu sumbu** — untuk source landscape (kasus paling umum, `sourceAspect > 9/16`), tinggi penuh dipertahankan dan cuma lebar yang di-crop, jadi cuma X yang bergerak (Y tetap 0); untuk source yang sudah portrait, sebaliknya. `computeCropDimensions()` selalu membulatkan ke angka genap (requirement `libx264`/`yuv420p`, menolak lebar/tinggi ganjil).

## Caption Styling (Fase 3 pasca-MVP)

Tiap `Clip` punya preset caption style (`CaptionStyle`: `DEFAULT` | `KARAOKE` | `BOLD_HIGHLIGHT`) yang menentukan cara `render-clip` men-styling subtitle burn-in-nya.

- **SRT diganti ASS/SSA sepenuhnya** (`apps/worker/src/subtitles.ts`'s `buildAss()`, menggantikan `buildSrt()` yang lama di `ffmpeg.ts`) — untuk **semua** preset termasuk `DEFAULT`, bukan cuma `KARAOKE`/`BOLD_HIGHLIGHT`, supaya cuma ada satu code path render (tidak ada percabangan format subtitle di `renderClip()`). `renderClip()`'s parameter di-rename `srtPath` -> `subtitlesPath` mengikuti perubahan ini. FFmpeg's `subtitles=` filter yang sama persis dipakai untuk ASS maupun SRT (libass auto-detect format dari isi file), jadi tidak ada perubahan di sisi `ffmpeg.ts` selain nama parameter.
- **`CaptionStyle` disimpan sebagai enum field langsung di `Clip`** (bukan model `CaptionStyle` terpisah) — preset tetap (bukan custom style bebas per-user), jadi kolom enum lebih dari cukup. Di-set lewat `PATCH /clips/:id` yang sudah ada dari Fase 1 (extend `UpdateClipDto`), **tidak auto re-render** — sama seperti trim manual, user klik render eksplisit lagi lewat `POST /clips/:id/render` (`ClipsService.update`/`.render`).
- **Word-level timestamp dari Whisper cuma berlaku ke depan, tidak di-backfill** — `transcribe.worker.ts` sekarang minta `timestamp_granularities: ['word', 'segment']` (satu API call yang sama, tidak nambah cost/panggilan terpisah) dan bucket tiap word ke segment yang memuat range waktunya, disimpan di `TranscriptSegment.words` (kolom `Json?`, nullable). Video yang sudah ditranskrip sebelum Fase 3 tidak punya data ini — kalau video begitu di-render dengan preset `KARAOKE`, `buildAss()` fallback ke teks polos per-segment yang word data-nya kosong (bukan gagal); tidak ada UI yang secara eksplisit mem-block pemilihan `KARAOKE` untuk video lama, cukup mengandalkan fallback graceful ini.
- **`KARAOKE` pakai fitur karaoke native ASS (`\k` tag)** — per-word `\k<centiseconds>` di dalam satu `Dialogue` line, cocok persis dengan kebutuhan "highlight kata yang sedang diucapkan secara sinkron" karena itu memang fitur bawaan format ASS untuk lirik karaoke. Perlu **dua `Style` ASS terpisah** (`Default` dan `Karaoke`, sama font/posisi, beda `PrimaryColour`/`SecondaryColour`) — `\k`'s pewarnaan (Secondary sebelum durasinya habis, Primary sesudahnya) berlaku ke `Style` yang direferensikan baris itu, dan baris **tanpa** tag `\k` selalu tampil di `PrimaryColour`-nya `Style` itu; kalau cuma ada satu `Style` bersama, preset `DEFAULT`/`BOLD_HIGHLIGHT` (tidak pakai `\k`) ikut kebagian warna aksen karaoke, bukan warna putih polos yang dimaksud. Segment `KARAOKE` yang word data-nya kosong pun sengaja dirender pakai `Style: Default` (bukan `Style: Karaoke`) supaya fallback-nya benar-benar netral, bukan solid warna aksen.
- **Heuristik keyword untuk `BOLD_HIGHLIGHT`, bukan input manual atau LLM** — kata yang mengandung angka/persentase, ALL-CAPS, atau diapit tanda kutip di-bold + diwarnai lewat override tag `{\b1\c...}...{\r}` inline di teks (`highlightKeywords()`). Ekstraksi keyword yang lebih pintar (semantik, bukan pattern-matching) ditunda ke Fase 5 (hook generator), yang memang sudah dalam roadmap untuk nambah LLM call baru. Beda dengan `KARAOKE`, preset ini **tidak butuh word-level timestamp** — cuma butuh teks segment biasa — jadi tetap berfungsi penuh untuk video lama.
- **Diverifikasi visual langsung** (bukan cuma unit test) — `buildAss()`'s output asli di-burn beneran lewat FFmpeg ke frame video test, dan hasilnya di-screenshot: kata angka/CAPS terbukti benar-benar bold+kuning untuk `BOLD_HIGHLIGHT`, dan highlight `KARAOKE` terbukti bergerak kata-per-kata seiring waktu (frame di t=3.1s/3.6s/4.3s masing-masing menunjukkan 1/2/3 kata yang sudah "ke-highlight").

## Observability (Fase 4 pasca-MVP)

`apps/api` dan `apps/worker` mengirim exception ke Sentry (`@sentry/node`) — sebelum fase ini, kegagalan job BullMQ cuma tercatat di log lokal container, tidak ada jejak terpusat yang gampang ditelusuri lintas deploy.

- **`SENTRY_DSN` opsional, bukan wajib** — beda dari `DATABASE_URL`/`STORAGE_*`/dst yang fail-fast di boot kalau kosong. `Sentry.init({ dsn: undefined })` cuma men-disable SDK-nya (semua `captureException` jadi no-op) tanpa throw, jadi aman dibiarkan kosong di dev lokal. Ditambahkan ke `apps/api/src/config/env.validation.ts` sebagai `@IsOptional()` (bukan required) dan didokumentasikan (bukan divalidasi, karena memang tidak perlu) di `apps/worker/src/env.ts`.
- **`apps/worker`: `Sentry.captureException(error, { tags: { videoId, clipId? } })`** di catch block masing-masing dari 3 job handler (`transcribe`, `detect-clips`, `render-clip`) — persis di titik yang sama dengan `console.error` yang sudah ada. Cuma tag ID yang dikirim, **tidak pernah** isi transcript, source video, atau `OPENAI_API_KEY`/`STORAGE_*` — sejalan dengan log `console.error` yang sudah ada sejak awal (cuma template string berisi ID). Fallback yang sudah "expected"/tidak menggagalkan job (mis. deteksi wajah gagal di Fase 2, fallback ke center-crop) **tidak** dilaporkan ke Sentry — cuma kegagalan yang benar-benar menggagalkan job yang masuk.
- **`apps/api`: `SentryExceptionFilter` (extends `BaseExceptionFilter`, `@Catch()` global)**, bukan integrasi otomatis `@sentry/nestjs` — cukup laporkan exception yang benar-benar tak terduga (bug), bukan full APM/route tracing yang tidak diminta fase ini. Filter ini cuma melapor ke Sentry untuk exception yang BUKAN `HttpException` **atau** `HttpException` dengan status >= 500 — `NotFoundException`/`BadRequestException`/dst (alur kontrol normal, bukan bug) sengaja tidak dilaporkan. Setelah lapor, selalu delegasi ke `super.catch()` supaya response HTTP (shape/status code) identik dengan sebelum filter ini ada — **hanya** exception object yang dikirim ke Sentry, tidak pernah request/response (yang bisa bawa cookie sesi/JWT).
- **`new SentryExceptionFilter(httpAdapter)` — bukan `HttpAdapterHost`-nya langsung** — `BaseExceptionFilter` manggil method macam `isHeadersSent()` langsung di objek yang dikasih ke constructor-nya, jadi butuh `app.get(HttpAdapterHost).httpAdapter` (adapter asli), bukan wrapper `HttpAdapterHost`-nya. Ini bug yang sempat kejadian beneran saat implementasi (lolos dari unit test yang nge-mock `BaseExceptionFilter`, ketahuan pas verifikasi E2E asli terhadap request 404/400 sungguhan) — pengingat kenapa exception filter yang di-mock unit test-nya tetap butuh dicoba beneran lewat request HTTP asli.
- **Sentry.init() dipanggil setelah titik yang sudah ada untuk load env**, bukan sebelumnya — `apps/worker/src/main.ts` sudah panggil `dotenv.config()` manual di baris paling atas (pola lama), jadi `initSentry()` taruh persis setelah itu. `apps/api` TIDAK pakai dotenv manual (`ConfigModule.forRoot()` yang urus env), jadi `initSentry()` dipanggil di `bootstrap()` setelah `NestFactory.create(AppModule)` selesai (titik pertama `process.env.SENTRY_DSN` dijamin sudah ke-populate) — bukan menambah `dotenv.config()` manual baru ke `apps/api` cuma demi Sentry, supaya tidak ada dua mekanisme load env yang tumpang tindih di app yang sama.

## Hook & Hashtag Generator (Fase 5 pasca-MVP)

`detect-clips`'s LLM call yang sudah ada (scoring virality) sekaligus menghasilkan `hookText` (saran baris pembuka ~3 detik pertama) dan `hashtags` (array tanpa `#`) per kandidat klip — bukan job/LLM call baru terpisah.

- **Extend prompt & JSON schema yang sudah ada, bukan job baru** — `RESPONSE_FORMAT`'s per-candidate schema nambah `hookText`/`hashtags` sebagai field required di samping `startTime`/`endTime`/`viralityScore` yang sudah ada; satu system prompt dan satu API call yang sama menghasilkan keduanya sekaligus. Dipilih supaya tidak nambah biaya+latency LLM call kedua untuk sesuatu yang bisa digabung sekali jalan (LLM sudah baca full transcript untuk scoring, konteks yang sama persis dibutuhkan untuk menulis hook/hashtag yang relevan).
- **Bahasa hookText/hashtags mengikuti bahasa transkrip video**, bukan dipaksa Inggris atau Indonesia — instruksi eksplisit di system prompt ("Write both hookText and hashtags in the same language as the transcript"), diverifikasi langsung end-to-end (transkrip video test berbahasa Inggris menghasilkan hook/hashtag berbahasa Inggris).
- **`hashtags` disanitasi di kedua sisi yang bisa menulisnya** — `sanitizeHashtags()` (`packages/shared/src/utils/hashtags.ts`, dipakai `detect-clips.worker.ts` dan `ClipsService.update`) strip leading `#` dan buang entry kosong/whitespace. LLM sudah diinstruksikan "no leading #", tapi tetap disanitasi (belt-and-suspenders, jangan percaya model sepenuhnya ikut instruksi format); demikian juga input manual lewat `PATCH /clips/:id` disanitasi dengan fungsi yang sama supaya data konsisten terlepas dari jalur mana yang menulisnya — bug ini sempat kejadian beneran (PATCH manual awalnya tidak disanitasi, ketahuan pas tes E2E asli mengirim `"#stripme"` dan hasilnya kesimpan dengan `#`-nya).
- **`Clip.hookText`/`Clip.hashtags` disimpan langsung di model `Clip`** (bukan tabel terpisah) — `hookText String?` (nullable; dalam praktiknya selalu terisi dari `detect-clips` karena field required di schema LLM, nullable cuma jaga-jaga) dan `hashtags String[] @default([])` (array native Postgres lewat Prisma, default array kosong bukan nullable supaya caller tidak perlu null-check). Migration baru (`add_clip_hook_and_hashtags`), sama pola dengan field `Clip` lain yang sudah ada.
- **Purely metadata, tidak memengaruhi hasil render** — beda dari `captionStyle` (Fase 3) yang benar-benar mengubah video hasil FFmpeg, `hookText`/`hashtags` tidak pernah dibaca oleh `render-clip`/`ffmpeg.ts`. Edit lewat `PATCH /clips/:id` (extend `UpdateClipDto` yang sudah ada dari Fase 1/3) langsung tersimpan tanpa perlu trigger render ulang sama sekali — beda dari trim/caption style yang tetap butuh klik "Render" eksplisit karena mengubah file video.
- **Ditampilkan di dua tempat**: read-only di dashboard (`apps/web/app/dashboard/page.tsx`'s clip list — "Clip Gallery") dan editable di Timeline Editor (`TimelineEditor.tsx`, pakai `setHookText`/`setHashtags` di `timelineStore.ts`, disimpan lewat tombol "Save" yang sama dengan trim/caption style). Input hashtags di editor pakai `<input>` **uncontrolled** (`key={clipId}` + `defaultValue` + commit di `onBlur`) bukan controlled biasa — controlled value yang di-derive ulang dari `hashtags.join(' ')` di tiap keystroke akan langsung membuang spasi/koma trailing yang baru diketik user, bikin tidak mungkin mulai kata kedua (ketahuan langsung waktu ditulis, sebelum sempat jadi bug nyata).

## Publish Center — Social Account Connect (Fase 6a pasca-MVP)

Fase 6 (publish scheduler multi-platform + analytics) dipecah jadi sub-fase; 6a cuma mengurus koneksi akun sosmed (connect/refresh/disconnect token) — **belum ada aksi publish sama sekali**. Ini sengaja diisolasi dulu karena OAuth per-platform adalah bagian paling berisiko/rumit dari seluruh Fase 6, dan lebih murah dibuktikan benar sendirian sebelum dibangun apa pun di atasnya.

- **YouTube Shorts dipilih sebagai platform pertama** (dari TikTok/Instagram Reels/YouTube Shorts) — OAuth2 Google standar, API paling matang, tidak ada gate app-review untuk scope upload dasar (beda dari TikTok's Content Posting API yang butuh audit dulu untuk direct-public-post, atau Instagram Reels yang butuh akun IG Business/Creator + flow publish 2 langkah via Meta Graph API). Prioritas platform berikutnya (Fase 6d) adalah keputusan bisnis, bukan teknis — lihat CLAUDE.md's breakdown history di percakapan, bukan didokumentasikan di sini.
- **Model baru `SocialAccount`** (`userId`, `platform` enum, `platformAccountId`, `displayName`, `accessToken`/`refreshToken` terenkripsi, `tokenExpiresAt`) — satu user bisa connect banyak akun, termasuk beberapa akun di platform yang sama (mis. 2 channel YouTube berbeda); unique constraint di `(userId, platform, platformAccountId)` supaya reconnect akun yang sama meng-upsert, bukan bikin baris duplikat.
- **Token dienkripsi app-level (AES-256-GCM), bukan mengandalkan enkripsi native provider hosting** — `apps/api/src/social/token-encryption.util.ts`, key dari env var baru `TOKEN_ENCRYPTION_KEY` (`openssl rand -hex 32`, sama pola generate dengan `JWT_SECRET`). Disimpan sebagai `"<iv>:<authTag>:<ciphertext>"` hex. Dipilih supaya portable ke provider hosting mana pun, tidak terikat fitur encryption-at-rest Postgres/cloud tertentu.
- **`GOOGLE_OAUTH_CLIENT_ID`/`GOOGLE_OAUTH_CLIENT_SECRET`/`TOKEN_ENCRYPTION_KEY` semua opsional di boot** (pola sama dengan `SENTRY_DSN` di Fase 4) — sisa `apps/api` harus tetap jalan normal buat semua orang yang belum setup Google Cloud OAuth client-nya sendiri. Konfigurasi yang kurang baru jadi error (503 "YouTube integration is not configured") persis di titik `GET /social/youtube/connect` benar-benar dipanggil, bukan bikin seluruh app gagal boot. `TOKEN_ENCRYPTION_KEY` yang kosong juga tidak diam-diam fallback ke key hardcoded manapun — `token-encryption.util.ts` sengaja throw error jelas di titik enkripsi/dekripsi pertama kali dipakai.
- **OAuth `state` param ditandatangani pakai `JwtModule` instance terpisah dari `AuthModule`** (`SocialModule` register `JwtModule.registerAsync()` sendiri, `JWT_SECRET` yang sama tapi expiry beda jauh lebih pendek — 10 menit) — dipakai murni buat mengaitkan callback dari Google balik ke user mana yang connect, bukan token sesi. `GET /social/youtube/callback` sengaja **tidak** dipasangi `JwtAuthGuard` — sumber kebenaran soal "siapa yang connect" adalah `state` yang di-decode, bukan cookie sesi (yang bisa saja sudah expired/logout di antara redirect ke Google dan baliknya).
- **`GET /social/youtube/connect` bukan `fetch()` dari `apps/web`, tapi `<a href>` (top-level browser navigation)** — OAuth butuh redirect browser sungguhan ke Google, yang tidak bisa dilakukan `fetch()`. Cookie sesi tetap terkirim di navigasi top-level ini (`SameSite=Lax` mengizinkan itu, dan `apps/web`/`apps/api` di dev sama-sama di bawah "site" `localhost`), jadi `JwtAuthGuard` tetap bisa resolve user di endpoint `connect` (beda dengan `callback`, lihat poin di atas).
- **`OAuth2Client` dari `google-auth-library`** (official Google client) dipakai untuk mekanisme OAuth (authorize URL, token exchange, refresh, revoke) — bukan hand-roll raw HTTP call ke endpoint OAuth Google, karena kesalahan kecil di kode OAuth (mis. parameter yang salah/kurang) adalah kelas bug yang mahal (auth diam-diam gagal atau lebih parah, celah keamanan). Fetch channel info (buat display name di UI) pakai `fetch()` polos ke YouTube Data API - satu endpoint sederhana, tidak perlu SDK `googleapis` yang jauh lebih berat cuma untuk itu.
- **Diverifikasi nyata sejauh yang bisa** — `apps/api` boot normal tanpa `GOOGLE_OAUTH_CLIENT_ID`/`SECRET`/`TOKEN_ENCRYPTION_KEY` sama sekali (state dev paling realistis sebelum user setup Google Cloud sendiri), `GET /social/accounts` mengembalikan `[]` yang benar, `GET /social/youtube/connect` mengembalikan 503 yang jelas. Seluruh lifecycle connect->reconnect(upsert)->list->deteksi-expiry->disconnect diverifikasi langsung terhadap Postgres asli dengan enkripsi AES-256-GCM asli (bukan mock). **Batasan yang diketahui**: round-trip OAuth sungguhan ke server Google (authorize -> consent -> exchange -> refresh -> revoke) tidak bisa diverifikasi end-to-end karena butuh kredensial OAuth client asli dari Google Cloud Console yang cuma bisa dibuat manual oleh user (sama seperti `SENTRY_DSN` di Fase 4) — mekanisme HTTP-nya sendiri (lewat `google-auth-library`) sudah diuji unit test dengan mock terhadap client library resminya.
- **Belum ada job BullMQ baru** — 6a murni operasi HTTP sinkron di `apps/api` (OAuth handshake), tidak menyentuh `apps/worker` sama sekali. `getValidAccessToken()` (proses refresh) ada sebagai method service yang teruji tapi belum dipanggil dari endpoint HTTP manapun — fase publish-clip berikutnya (6b) yang akan benar-benar memakainya.

## Publish Center — Manual "Publish Now" (Fase 6b pasca-MVP)

Tombol "Publish to YouTube" di dashboard (Clip Gallery) untuk klip yang sudah selesai di-render — meng-upload klip itu sungguhan ke channel YouTube user yang sudah connect (Fase 6a), lewat job BullMQ baru `publish-clip`. Publish selalu langsung begitu tombol diklik di fase ini — lihat "Publish Center — Scheduling (Fase 6c pasca-MVP)" di bawah untuk publish di waktu yang dijadwalkan.

- **`packages/social` (package baru)** — OAuth client (`YouTubeOAuthClient`), enkripsi token (`encryptToken`/`decryptToken`), `resolveAccessToken()` (cek expiry + refresh kalau perlu), dan `uploadYouTubeVideo()` dipindah/ditaruh di sini, bukan di `apps/api/src/social/` seperti Fase 6a. Alasannya: `publish-clip` job di `apps/worker` butuh access token YouTube yang valid (decrypt + refresh), tapi `apps/worker` tidak bisa import langsung dari `apps/api` (dua deployable app terpisah, cuma `packages/*` yang shared antar keduanya). Ini pengecualian sadar terhadap prinsip "ekstrak ke shared code di duplikasi ke-3, bukan ke-2" yang biasa dipakai proyek ini — OAuth/enkripsi token adalah kelas bug berisiko tinggi (drift halus di sini = celah keamanan atau job publish yang diam-diam rusak), jadi lebih murah dibagi sejak awal daripada didupilkasi sekali dulu. `apps/api`'s `SocialAccountsService`/`SocialController` sekarang import dari `@speedora/social`, bukan file lokal lagi.
- **`googleapis` dipakai untuk `videos.insert` yang sesungguhnya** (bukan hand-roll protokol resumable upload) — konsisten dengan keputusan Fase 6a pakai `google-auth-library` resmi untuk OAuth ("implementasi custom yang subtly-wrong untuk protokol multi-request itu kelas bug yang jauh lebih mahal daripada bobot dependency-nya"). **`googleapis` membundel salinan internal `google-auth-library` sendiri** yang tipe `OAuth2Client`-nya secara struktural tidak kompatibel dengan package `google-auth-library` standalone yang dipakai `youtube-oauth.client.ts` untuk alur connect/refresh/revoke — walau keduanya resolve ke versi yang sama-sama valid secara semver. Solusinya bukan memaksa satu helper `authorizedClient()` dipakai bersama, tapi memisahkan: `youtube-upload.client.ts` (upload) membangun objek auth-nya sendiri lewat `google.auth.OAuth2` dari `googleapis`'s namespace sendiri, sementara `youtube-oauth.client.ts` (connect/refresh/revoke) tetap pakai `OAuth2Client` standalone — dua konstruksi terpisah yang masing-masing konsisten secara tipe dengan library yang dia pakai.
- **Model baru `PublishRecord`** (`clipId`, `socialAccountId`, `status` enum `QUEUED`/`PUBLISHING`/`PUBLISHED`/`FAILED`, `platformPostId`, `errorMessage`, `publishedAt`, plus `scheduledAt DateTime?` yang sengaja ditambahkan sekarang walau selalu `null` di 6b — supaya Fase 6c (scheduling) tidak butuh migration kedua cuma untuk kolom itu) — satu baris per **percobaan** publish klip ke satu akun sosmed (retry BullMQ mengubah status baris yang sama, bukan bikin baris baru). Relasi ke `Clip` dan `SocialAccount`, keduanya `onDelete: Cascade`.
- **`ClipsService.publish()` bikin baris `PublishRecord` secara sinkron (status `QUEUED`) sebelum enqueue job** — sama seperti `render()`'s pola "clear `outputUrl` dulu sebelum enqueue" dari Fase 1: supaya UI langsung bisa nampilin status "Queued to publish..." begitu tombol diklik, tanpa nunggu job benar-benar mulai diproses worker.
- **Job `publish-clip` cuma bawa `{ publishRecordId }` di payload-nya**, bukan snapshot clipId/socialAccountId/dll — konsisten dengan pola job lain di proyek ini (payload minimal, re-fetch state terkini dari Postgres di worker) supaya tidak ada risiko data enqueue-time yang sudah basi dipakai saat job benar-benar jalan. Worker fetch `PublishRecord` beserta relasi `clip`+`socialAccount`-nya sekaligus lewat satu `include`.
- **`publish-clip` adalah job BullMQ pertama di proyek ini yang pakai retry otomatis bawaan BullMQ** (`attempts: 3`, `backoff: { type: 'exponential', delay: 30_000 }`, dikonfigurasi di `ClipsService.publish()` sebagai argumen ketiga `queue.add()`) — beda dari `transcribe`/`detect-clips`/`render-clip` yang semuanya gagal sekali lalu nunggu user klik Retry manual. Dibedakan sengaja karena panggilan ke API platform eksternal (YouTube) punya kelas kegagalan transient yang tidak dimiliki job-job lain (rate limit, 5xx sementara) — user tidak perlu tahu/ikut campur soal itu. `PublishRecord.status` cuma di-set ke `FAILED` (dengan `errorMessage`) di attempt **terakhir** (`job.attemptsMade + 1 >= job.opts.attempts`) — kalau masih ada retry tersisa, baris dibiarkan `PUBLISHING` supaya UI tidak salah nampilin "gagal" padahal masih ada percobaan otomatis berikutnya yang jalan.
- **Default `privacyStatus: 'unlisted'`** untuk video yang di-upload — ini video sungguhan yang benar-benar ke-publish ke channel YouTube user begitu tombol "Publish now" diklik. `public` langsung dianggap terlalu berisiko untuk fase yang tujuannya "membuktikan jalur upload-ke-platform bekerja" (klip yang salah pilih atau caption yang belum pas bisa langsung live tanpa jaring pengaman), sementara `private` dianggap mengalahkan tujuan "publish now" itu sendiri (user tetap harus ubah manual visibility-nya lewat YouTube Studio). Tidak ada opsi visibility yang bisa diubah user di fase ini — keputusan produk untuk fase berikutnya kalau dibutuhkan.
- **Title dari `Clip.hookText`** (fallback ke `"Clip <id>"` kalau kosong) dan **description dari `Clip.hashtags`** (di-join jadi `#tag1 #tag2 ...`) — field yang sama yang sudah ada dari Fase 5, tidak ada input title/description terpisah untuk publish di fase ini.
- **`GET /videos/:id` dan `PATCH/POST /clips/:id/*` sekarang ikut mengembalikan `Clip.publishRecords[]`** (`toSharedPublishRecord()` di `apps/api/src/social/publish-record.util.ts`, dipakai `ClipsService` dan `VideosService`) — `platform` di-denormalisasi dari `SocialAccount` yang di-join supaya dashboard bisa langsung nampilin "Published to YouTube" tanpa query terpisah. Dashboard polling `GET /videos` yang sudah ada (interval 2 detik, sama seperti transcribe/detect-clips/render-clip) otomatis kepakai buat mantau progres publish juga — tidak ada endpoint/mekanisme polling baru.
- **Diverifikasi nyata sejauh yang bisa** — lifecycle penuh `PublishRecord` (create -> QUEUED -> job jalan -> PUBLISHING -> PUBLISHED/FAILED) diuji lewat mock jaringan (`uploadYouTubeVideo`/`resolveAccessToken` di-mock) terhadap Postgres asli, termasuk retry/backoff yang benar-benar terpasang di job BullMQ yang di-enqueue dan tagging Sentry (`publishRecordId`/`clipId`/`socialAccountId` saja, tidak pernah token atau isi video). **Batasan yang diketahui**: panggilan `videos.insert` sungguhan ke server YouTube (upload video asli, konfirmasi visibility `unlisted` benar-benar diterapkan di sisi Google) tidak bisa diverifikasi end-to-end tanpa kredensial OAuth + akun YouTube asli milik user — sama seperti batasan OAuth round-trip di Fase 6a.

## Publish Center — Scheduling (Fase 6c pasca-MVP)

`POST /clips/:id/publish` menerima `scheduledAt` (ISO 8601) opsional — kalau diisi (dan waktunya di masa depan), klip dijadwalkan publish nanti alih-alih langsung, lewat `PublishRecord` status baru `SCHEDULED` dan job BullMQ repeatable baru `schedule-publish-clip` di `apps/worker` yang men-poll Postgres.

- **Mekanisme: BullMQ repeatable job yang polling Postgres, bukan BullMQ delayed job** — dipilih secara eksplisit di antara dua kandidat karena konsisten dengan prinsip yang sudah ada di proyek ini ("PostgreSQL sebagai source of truth, Redis cuma antrian/cache, bukan penyimpanan permanen" — lihat "Keputusan Arsitektur" di bawah). Kalau "jadwal" disimpan sebagai delayed job di dalam Redis (opsi lain yang dipertimbangkan), kehilangan data Redis (restart tanpa persistence, flush, dst) bisa diam-diam menghapus klip yang harusnya publish di masa depan tanpa jejak apa pun. Dengan polling, `PublishRecord.scheduledAt` di Postgres tetap jadi satu-satunya sumber kebenaran — Redis kehilangan data cuma menunda job poller berikutnya, tidak pernah menghapus rencana publish. Trade-off yang diterima: presisi waktu publish maksimal mundur sejauh interval polling (`POLL_INTERVAL_MS` = 60 detik, `apps/worker/src/workers/schedule-publish-clip.worker.ts`).
- **`PublishStatus` nambah `SCHEDULED`** (state paling awal, sebelum `QUEUED`) — dipakai murni untuk klip yang scheduledAt-nya belum tiba. `ClipsService.publish()` sekarang bercabang: `scheduledAt` kosong -> tetap seperti Fase 6b (`status: QUEUED`, langsung `publishClipQueue.add()`); `scheduledAt` terisi (divalidasi harus di masa depan, kalau tidak `BadRequestException`) -> `status: SCHEDULED`, **tidak** enqueue apa pun ke `publish-clip` sampai poller yang men-trigger-nya nanti.
- **Poller meng-klaim baris secara atomik** — `schedule-publish-clip.worker.ts`'s processor query semua `PublishRecord` dengan `status: SCHEDULED AND scheduledAt <= now()`, lalu untuk tiap baris jalankan `updateMany({ where: { id, status: SCHEDULED }, data: { status: QUEUED } })` dan cuma `publishClipQueue.add()` kalau `count === 1`. Guard `status: SCHEDULED` di `WHERE` klausa `updateMany` inilah yang membuat klaim atomik — walau dua firing job repeatable ini entah bagaimana tumpang tindih (mis. firing sebelumnya masih jalan lambat pas firing berikutnya mulai), baris yang sama tidak akan pernah ke-enqueue dua kali, karena `updateMany` kedua yang mencoba baris yang sudah bukan `SCHEDULED` lagi akan `count: 0`.
- **`PUBLISH_RETRY_OPTIONS` dipindah ke `packages/shared`** (dari const privat di `ClipsService`, Fase 6b) — begitu `apps/worker`'s poller juga perlu men-enqueue `publish-clip` dengan retry/backoff yang **persis sama** dengan `ClipsService.publish()`'s immediate-publish path, duplikasi angka retry di dua tempat (rawan drift diam-diam) jadi lebih mahal daripada satu konstanta shared — pola yang sama dengan `filterSegmentsForClip`/`sanitizeHashtags` sebelumnya.
- **`apps/worker` sekarang juga punya `Queue` producer untuk `PUBLISH_CLIP`** (`apps/worker/src/queues.ts`'s `publishClipQueue`) — di Fase 6b, `publish-clip.worker.ts` adalah leaf job (tidak self-chain ke job lain), jadi `apps/worker` sebelumnya tidak pernah perlu jadi *producer* ke queue itu, cuma consumer. Poller adalah producer pertama di sisi `apps/worker` untuk `PUBLISH_CLIP`.
- **Cancel & reschedule** (`DELETE`/`PATCH /clips/:id/publish/:recordId`) **cuma berlaku selagi `SCHEDULED`** — begitu poller sempat meng-klaimnya (jadi `QUEUED` atau lebih lanjut), request ini dianggap "tidak ditemukan" (404), bukan mencoba membatalkan/menghentikan job yang sudah di tangan worker. Scoping `id + clipId + status: SCHEDULED` dicek atomik dalam satu `deleteMany`/`updateMany` (bukan baca-lalu-tulis terpisah) supaya tidak ada race antara pengecekan status dan aksinya sendiri.
- **Cancel meng-hapus baris `PublishRecord`-nya langsung** (bukan status `CANCELED` baru) — baris `SCHEDULED` yang dibatalkan belum pernah benar-benar "terjadi" (belum di-enqueue sama sekali), jadi tidak ada nilai historis untuk disimpan; menambah satu enum value lagi cuma untuk kasus ini dianggap scope creep dibanding sekadar menghapus baris rencana yang belum jalan.
- **UI dashboard**: input `datetime-local` + tombol "Schedule" di samping "Publish now" yang sudah ada. Baris `PublishRecord` berstatus `SCHEDULED` ditampilkan beda dari status lain — "Scheduled for \<tanggal\>" plus tombol Cancel dan Reschedule (input `datetime-local` inline) — sementara status lain (`QUEUED`/`PUBLISHING`/`PUBLISHED`/`FAILED`) tetap pakai tampilan label singkat yang sudah ada dari Fase 6b. Dashboard's polling `GET /videos` yang sudah ada (2 detik) otomatis menangkap transisi `SCHEDULED` -> `QUEUED` begitu poller men-klaimnya, tanpa mekanisme polling baru.
- **Diverifikasi lewat unit test terhadap logic klaim atomik dan validasi tanggal** (mock Prisma/BullMQ) — belum ada verifikasi E2E baru terhadap Postgres+Redis asli khusus untuk scheduling di luar yang sudah dicakup Fase 6b's publish-clip pipeline (poller ini cuma menambah satu langkah `SCHEDULED -> QUEUED` sebelum job yang sama persis berjalan).

## Publish Center — Platform Kedua: TikTok (Fase 6d pasca-MVP)

Tombol "Connect TikTok" di halaman `/accounts` dan opsi platform TikTok di tombol "Publish to \<platform\>" dashboard — `SocialAccount.platform` nambah `TIKTOK`, dan `publish-clip.worker.ts` bercabang upload ke YouTube atau TikTok tergantung platform akun yang dipilih.

- **Mode "Upload to Inbox" (draft), bukan Direct Post — keputusan bisnis eksplisit, bukan default teknis.** TikTok Content Posting API punya dua target publish: Direct Post (publish sungguhan ke profil user, tapi butuh app di-audit dulu oleh TikTok — approval tidak dijamin dan bisa makan waktu) dan Upload to Inbox (kirim video sebagai draft ke inbox TikTok user, jalan langsung tanpa audit, tapi user tetap harus buka app TikTok dan post manual untuk benar-benar live). Proyek ini memilih Upload to Inbox supaya fitur publish TikTok bisa langsung dipakai tanpa bergantung pada approval TikTok yang tidak pasti — konsekuensinya, **"Publish to TikTok" TIDAK benar-benar mem-publish** seperti "Publish to YouTube"; UI secara eksplisit menampilkan "Sent to TikTok — open the TikTok app to finish posting" (bukan "Published") untuk `PublishRecord` TikTok yang statusnya `PUBLISHED`, supaya user tidak salah kira klipnya sudah live.
- **Tidak ada SDK Node resmi untuk TikTok** (beda dari Google yang punya `google-auth-library`/`googleapis`) — `tiktok-oauth.client.ts` dan `tiktok-upload.client.ts` (keduanya di `packages/social`) hand-roll seluruh HTTP call lewat `fetch()` polos. Ini BUKAN pelanggaran terhadap prinsip "pakai official client library" dari Fase 6a/6b — prinsip itu berlaku ketika ada library resmi untuk dihindari hand-roll-nya; untuk TikTok, tidak ada alternatif sama sekali.
- **`resolveAccessToken()` digeneralisasi** — sebelumnya (Fase 6b) parameternya secara eksplisit bertipe `YouTubeOAuthClient`. Sekarang menerima `OAuthRefreshClient`, interface minimal (`refreshAccessToken(refreshToken): Promise<{accessToken, refreshToken, expiresAt}>`) yang diimplementasikan baik `YouTubeOAuthClient` maupun `TikTokOAuthClient` — supaya logic "cek expiry, refresh kalau perlu" yang sama tetap dipakai bersama lintas platform, bukan didupilkasi atau di-`as any`-kan.
- **`SocialAccountsService` dan `publish-clip.worker.ts` masing-masing punya `clientFor(platform)`/`oauthClientFor(platform)`** — satu `switch` kecil yang memetakan `SocialAccount.platform` ke instance client OAuth yang tepat (`YouTubeOAuthClient` atau `TikTokOAuthClient`), dipakai oleh `disconnect()` (revoke), `getValidAccessToken()`/`resolveAccessToken()` (refresh), dan upload. Dua tempat, bukan diekstrak ke `packages/social` — sesuai konvensi "ekstrak di duplikasi ke-3" proyek ini, switch 3 baris ini terlalu kecil untuk dijadikan abstraksi bersama.
- **Scope OAuth cuma `user.info.basic` + `video.upload`** (bukan `video.publish`, yang butuh audit "direct public post") — `video.upload` adalah scope yang benar untuk Upload to Inbox dan tersedia untuk app yang belum diaudit (walau masih ada batasan sandbox: akun TikTok penguji biasanya perlu didaftarkan sebagai "target user" di TikTok Developer Portal selama app belum lolos App Review dasar).
- **Upload TikTok single-chunk, buffer penuh di memori** (`total_chunk_count: 1`, bukan protokol multi-chunk PUT TikTok yang lengkap) — klip di proyek ini dibatasi ~60 detik oleh prompt LLM `detect-clips` (Fase MVP), jadi ukurannya selalu jauh di bawah batas satu PUT. Endpoint init TikTok butuh `video_size` diketahui persis di awal (beda dari `uploadYouTubeVideo()` yang bisa pipe stream langsung ke `googleapis`), jadi stream harus di-buffer penuh dulu sebelum panggilan init.
- **Tidak ada title/caption yang bisa diisi via API untuk mode Inbox** — beda dari YouTube (`hookText`/`hashtags` dipetakan ke title/description), endpoint `inbox/video/init` TikTok cuma menerima `source_info`, tidak ada field caption sama sekali di mode ini (cuma tersedia di Direct Post). User yang mengisi caption sendiri saat menyelesaikan posting di app TikTok — `hookText`/`hashtags` klip tidak dipakai sama sekali untuk publish TikTok.
- **Diverifikasi lewat unit test terhadap OAuth client, upload client, dan dispatch platform** (mock `fetch`/Prisma/BullMQ) — sama seperti batasan Fase 6a/6b, round-trip OAuth sungguhan ke server TikTok dan upload video asli ke inbox TikTok tidak bisa diverifikasi end-to-end tanpa TikTok Developer app + akun TikTok asli milik user.

## Publish Center — Platform Ketiga: Instagram Reels (Fase 6d follow-up)

Tombol "Connect Instagram" di `/accounts` dan opsi platform Instagram di tombol "Publish to \<platform\>" dashboard — `SocialAccount.platform` nambah `INSTAGRAM`. Beda dari YouTube/TikTok, Instagram punya dua kendala arsitektur baru yang keduanya keputusan eksplisit (bukan default teknis semata): jalur OAuth-nya, dan cara klip diserahkan ke server Meta.

- **Jalur OAuth: Facebook Login + akun IG ditautkan ke Facebook Page (bukan Instagram Login langsung yang lebih baru dari Meta)** — dipilih karena lebih matang/terdokumentasi, walau konsekuensinya user WAJIB punya akun Instagram Business/Creator yang ditautkan ke sebuah Facebook Page yang dia kelola (bukan akun IG personal biasa). Sama seperti TikTok, scope `instagram_content_publish` produksi penuh butuh App Review dari Meta; sebelum lolos review, cuma akun yang didaftarkan sebagai Instagram Tester di App Meta yang bisa dipakai.
- **Video diserahkan ke Meta lewat presigned URL langsung dari R2/S3, bukan byte-upload seperti YouTube/TikTok** — Instagram Content Publishing API TIDAK punya opsi terima upload byte sama sekali; API-nya cuma bisa "ambil video dari URL publik yang bisa diakses tanpa login" (`video_url` di body `POST /{ig-user-id}/media`). Ini pertama kalinya proyek ini mengekspos sesuatu yang menyerupai link langsung ke bucket — sebelumnya SEMUA akses klip lewat endpoint `apps/api` yang butuh sesi login (lihat "Keputusan Arsitektur" — storage tidak pernah diekspos publik). `packages/storage`'s `getPresignedDownloadUrl(key, expiresInSeconds)` (pakai `@aws-sdk/s3-request-presigner`) generate URL sementara (15 menit, `INSTAGRAM_PRESIGNED_URL_TTL_SECONDS` di `publish-clip.worker.ts`) yang acak/tidak bisa ditebak dan otomatis kedaluwarsa — dipertimbangkan lebih aman/lebih sederhana daripada alternatif "endpoint publik baru di `apps/api` dengan token sekali-pakai" yang juga sempat dipertimbangkan.
- **Model refresh token Facebook beda dari YouTube/TikTok, tapi tetap muat di interface `OAuthRefreshClient` yang sama** — Meta tidak punya refresh_token terpisah untuk alur ini; yang ada cuma access token USER long-lived (~60 hari) yang di-re-exchange lewat grant `fb_exchange_token` buat diperpanjang. Tapi token yang benar-benar dipakai untuk panggilan Content Publishing API adalah **Page access token** (didapat dari `/me/accounts`, bukan token user itu sendiri) — jadi `SocialAccount.accessToken` menyimpan Page token, dan `SocialAccount.refreshToken` menyimpan token USER long-lived (dipakai `refreshAccessToken()` buat re-derive Page token baru nanti). `InstagramOAuthClient.refreshAccessToken(refreshToken)` tetap mengembalikan bentuk `{accessToken, refreshToken, expiresAt}` yang sama seperti YouTube/TikTok, jadi `resolveAccessToken()` tidak perlu tahu bedanya sama sekali.
- **`fetchAccountInfo()` otomatis pilih Facebook Page PERTAMA yang punya akun Instagram Business tertaut** — user dengan banyak Page/akun IG belum bisa memilih lewat UI picker di fase ini; simplifikasi yang sama semangatnya dengan keputusan "klaim job repeatable pertama" di Fase 6c, didokumentasikan sadar sebagai batasan, bukan bug.
- **Container create -> poll status -> publish, dua panggilan API terpisah dengan proses async di antaranya** — beda dari YouTube (`videos.insert` satu panggilan sinkron) dan TikTok (init + PUT satu rangkaian cepat), Instagram butuh `POST /{ig-user-id}/media` (bikin container), lalu **polling** `GET /{container-id}?fields=status_code` sampai `FINISHED` (maksimal 5 menit, `POLL_TIMEOUT_MS` di `instagram-upload.client.ts`) sebelum `POST /{ig-user-id}/media_publish` benar-benar bisa jalan. `ERROR`/`EXPIRED` dianggap gagal permanen (tidak di-retry di dalam polling itu sendiri — BullMQ's retry/backoff yang sudah ada di level job yang menangani percobaan ulang).
- **Caption Instagram Reels beda dari YouTube's title+description terpisah** — cuma ada satu field `caption`, jadi `hookText` dan `hashtags` digabung (`buildCaption()`, pisah baris kosong) alih-alih dipetakan ke dua field terpisah seperti YouTube. Beda juga dari TikTok Upload-to-Inbox yang sama sekali tidak punya field caption — Instagram Reels justru API publish sungguhan (`media_publish` yang berhasil = benar-benar live), jadi `hookText`/`hashtags` dipakai penuh, bukan diabaikan.
- **"Published" untuk Instagram artinya publish sungguhan** (beda dari TikTok's "Upload to Inbox") — tidak ada override teks khusus di dashboard seperti TikTok, `PUBLISHED` + `platformPostId` (media id Instagram) ditampilkan apa adanya. Tidak ada link ke Reel-nya langsung (media id numerik dari `media_publish` bukan permalink/shortcode publik — mengambil permalink butuh satu panggilan Graph API tambahan, ditunda ke fase berikutnya kalau dibutuhkan).
- **Diverifikasi lewat unit test terhadap OAuth client (termasuk model refresh 2-langkah), upload client (container create + polling + publish, pakai fake timer buat polling interval), dispatch platform, dan `getPresignedDownloadUrl`** (mock `fetch`/`@aws-sdk/s3-request-presigner`/Prisma/BullMQ) — sama seperti batasan YouTube/TikTok, round-trip OAuth sungguhan ke server Meta dan publish Reel asli tidak bisa diverifikasi end-to-end tanpa Meta app + akun Instagram Business/Facebook Page asli milik user.

## Premium Transcription — Groq (default) + OpenAI Whisper (Fase 7 pasca-MVP)

Layar baru "Pilih Mesin Transkripsi" (`apps/web`'s `EngineChoice` component) muncul sebelum layar upload video/import YouTube yang sudah ada — user memilih **Groq Whisper large-v3-turbo** (gratis, default) atau **OpenAI Whisper** (premium, gerbang berbayar sungguhan lewat Midtrans) untuk video yang akan diupload/diimpor.

- **Dipilih per-video, bukan setting akun** — `apps/web`'s `provider` state di-reset ke `null` setiap kali satu siklus upload/import selesai (`handleUploadAnother`), jadi `EngineChoice` selalu tampil lagi sebelum video berikutnya. Konsisten dengan keputusan produk eksplisit: user bisa pakai Groq untuk satu video dan OpenAI untuk video berikutnya tanpa perlu ubah setting apa pun.
- **"Premium" adalah gerbang berbayar sungguhan, bukan cuma label UI** — `TranscriptionProvider.OPENAI` cuma bisa dipakai kalau user punya `PremiumCredit` berstatus `PAID` yang belum dipakai (`videoId` masih `null`). `VideosService.upload()`/`.importFromYoutube()` menolak (400) permintaan OPENAI kalau tidak ada kredit tersedia — dicek dulu (murah, sebelum sentuh storage/enqueue apa pun), lalu diklaim secara atomik (`PaymentsService.consumeCredit()`, `updateMany` dengan guard `videoId: null` di `WHERE`) begitu video row-nya dibuat. Race yang kalah klaim (jarang, tapi mungkin) di-rollback: video yang baru dibuat dihapus (plus object storage-nya kalau sudah sempat diupload) sebelum melempar 400 — tidak pernah membiarkan video OPENAI "menggantung" tanpa kredit di belakangnya.
- **Bayar per-video (pay-per-use), bukan langganan** — Rp 10.000/video (`PREMIUM_TRANSCRIPTION_PRICE_IDR`, `packages/shared/src/types/payment.ts`), satu transaksi Midtrans Snap = satu `PremiumCredit` yang bisa dipakai sekali (untuk satu video, lalu `videoId`-nya terisi permanen). Dipilih atas langganan karena user mungkin cuma butuh kualitas premium untuk sebagian kecil video, bukan semuanya.
- **Midtrans Snap dipilih sebagai payment gateway** (bukan Stripe/lainnya) — dukungan metode pembayaran Indonesia (QRIS, virtual account bank lokal, e-wallet) jauh lebih lengkap daripada gateway global untuk use-case ini. `midtrans-client` (npm resmi) dipakai untuk membuat transaksi Snap (`Snap.createTransaction()`) di `apps/api/src/payments/payments.service.ts`, bukan hand-roll HTTP call — konsisten dengan prinsip "pakai official client library kalau ada" dari Fase 6a/6b.
- **Model baru `PremiumCredit`** (`userId`, `status` enum `PENDING`/`PAID`/`FAILED`/`EXPIRED`, `amount`, `midtransOrderId` unik, `videoId` unik nullable) — satu baris per transaksi Midtrans. `videoId` nullable + unique berarti "kredit yang sudah PAID tapi `videoId` masih null" = kredit yang tersedia dipakai; `PaymentsService.getAvailability()` query persis kondisi itu. `Video` nambah kolom `transcriptionProvider` (enum `GROQ`/`OPENAI`, default `GROQ`) yang di-set sekali saat video dibuat dan tidak pernah berubah — dipakai `resolveWhisperClient()` (`apps/worker/src/workers/transcribe.worker.ts`) untuk memilih client Whisper yang benar, dan diteruskan langsung di job payload (`TranscribeJobData.provider`) alih-alih di-query ulang, sama pola dengan `sourceUrl`.
- **Groq dipakai lewat endpoint OpenAI-compatible-nya** (`https://api.groq.com/openai/v1`, model `whisper-large-v3-turbo`) — SDK `openai` resmi dipakai apa adanya untuk kedua provider (`apps/worker/src/groq.ts` cuma override `baseURL`/`apiKey`), bukan SDK terpisah, karena Groq memang didesain drop-in compatible dengan API OpenAI untuk endpoint audio transcription ini.
- **`GROQ_API_KEY` wajib di boot, `OPENAI_API_KEY` opsional** — kebalikan dari sebelumnya (`OPENAI_API_KEY` dulu wajib). Groq sekarang jalur default/gratis yang dipakai semua orang, jadi harus ada sejak awal; OpenAI cuma dibutuhkan kalau ada user yang benar-benar bayar premium, jadi worker tetap boot normal dan melayani semua transkripsi gratis walau `OPENAI_API_KEY` belum diset — cuma job OPENAI yang gagal jelas (`resolveWhisperClient()` throw sebelum kerja apa pun dimulai) di titik itu.
- **Webhook Midtrans (`POST /payments/webhook/midtrans`) adalah satu-satunya sumber kebenaran soal status bayar** — bukan callback `onSuccess`/`onPending` Snap.js sisi client (`apps/web`), yang cuma dipakai untuk memicu polling `GET /payments/premium-transcription/status`, bukan langsung dipercaya. Endpoint webhook sengaja **tidak** dipasangi `JwtAuthGuard` (ini panggilan server-to-server dari Midtrans, bukan browser dengan cookie sesi) — kepercayaannya sepenuhnya dari verifikasi signature (`sha512(order_id + status_code + gross_amount + MIDTRANS_SERVER_KEY)`, `midtrans-signature.util.ts`, dibandingkan pakai `crypto.timingSafeEqual`). `PremiumCredit` diklaim dari `PENDING` ke status baru lewat `updateMany` dengan guard `status: PENDING` di `WHERE` (bukan `update` polos) — Midtrans bisa (dan memang) mengirim notifikasi yang sama lebih dari sekali, jadi update ini harus idempotent dan tidak pernah menimpa balik kredit yang sudah `PAID` (mungkin sudah terpakai, `videoId`-nya sudah terisi).
- **`MIDTRANS_SERVER_KEY`/`MIDTRANS_CLIENT_KEY`/`MIDTRANS_IS_PRODUCTION` opsional di boot** (pola sama dengan `GOOGLE_OAUTH_CLIENT_ID` dkk di Fase 6a) — `POST /payments/premium-transcription/checkout` dan webhook cuma 503 (`MidtransNotConfiguredError`) sampai di-set, sisa app (termasuk jalur gratis Groq) tetap jalan normal. `NEXT_PUBLIC_MIDTRANS_CLIENT_KEY` (apps/web, nilai yang sama dengan `MIDTRANS_CLIENT_KEY` sisi server — client key Midtrans memang didesain aman diekspos ke browser) dipakai Snap.js; kosong berarti tombol "Bayar & Pakai OpenAI" gagal dengan pesan jelas alih-alih Snap.js embed yang rusak diam-diam.
- **Diverifikasi lewat unit test terhadap signature verification (termasuk constant-time comparison), `PaymentsService` (checkout, availability, webhook state machine, atomic credit claim/rollback termasuk kasus race), dan wiring `VideosService`** (mock `midtrans-client`/Prisma/BullMQ) — sama seperti batasan integrasi eksternal lain di proyek ini (OAuth Fase 6a/6b/6d, dst), pembayaran sungguhan lewat server Midtrans asli (checkout beneran, webhook asli, saldo asli berpindah) tidak bisa diverifikasi end-to-end tanpa akun Midtrans (sandbox pun cukup) yang cuma bisa dibuat manual oleh user — lihat README untuk cara mendapatkan kredensial sandbox-nya.

## Content Intelligence — Multi-Metric Scoring & Explainable AI (Fase 8 pasca-MVP)

`detect-clips`'s satu-satunya LLM call (yang sudah menghasilkan `viralityScore`/`hookText`/`hashtags` sejak Fase 5) diperluas untuk juga menghasilkan breakdown multi-metrik, alasan pemilihan klip yang bisa dibaca manusia (explainable AI), dan klasifikasi konten (topik, kata kunci, intent, CTA) — bukan job/LLM call baru terpisah, dengan alasan yang sama seperti Fase 5: model sudah membaca full transcript untuk scoring, konteks yang sama persis dibutuhkan untuk analisis-analisis ini.

- **Kejujuran skala: heuristik LLM, bukan model prediksi terlatih** — `ClipScores` (`hookStrength`, `educationalValue`, `curiosity`, `emotion`, `storytelling`, `novelty`, `trustAuthority`, masing-masing 0-100) adalah estimasi GPT dari membaca transcript, persis seperti `viralityScore` yang sudah ada sejak awal — **bukan** metrik yang dilatih/dikalibrasi dari data engagement sungguhan. Keputusan sadar untuk **tidak** membangun fitur seperti "Retention Prediction"/"Scroll Stop Probability"/"Share Probability" sebagai angka prediksi terpisah — proyek ini tidak punya dataset engagement untuk melatih model semacam itu, dan menyajikan tebakan LLM sebagai "prediksi terlatih" akan menyesatkan pengguna. `reason` (1-2 kalimat, field baru) adalah wujud nyata dari "explainable AI" yang diminta — ditulis eksplisit untuk dibaca manusia ("kenapa klip ini dipilih"), bukan sekadar angka opaque.
- **`scores` disimpan sebagai satu kolom JSON** (`Clip.scores`, mirip `TranscriptSegment.words`) — bukan satu kolom Float per metrik — karena selalu dibaca/ditulis sebagai satu unit dan penambahan metrik baru di fase berikutnya tidak perlu migration baru. `topics`/`keywords` tetap `String[]` biasa (konsisten dengan `hashtags` yang sudah ada); `intent`/`ctaText`/`reason` adalah `String?` polos.
- **`intent` adalah string bebas yang dibatasi ke set tetap** (`educate`/`entertain`/`persuade`/`inspire`/`story`/`other`, lihat `detect-clips.worker.ts`'s `INTENTS`) — bukan enum Postgres seperti `CaptionStyle`, karena ini klasifikasi konten oleh LLM (bisa perlu nilai baru tanpa migration schema), bukan preset render yang dikontrol kode.
- **Response schema OpenAI structured output diperluas**, bukan LLM call kedua — `RESPONSE_FORMAT`'s per-candidate JSON schema nambah `scores` (nested object, semua 7 metrik `required`), `reason`, `topics`, `keywords`, `intent` (pakai `enum` JSON Schema, divalidasi strict oleh OpenAI sendiri), `ctaText`. Semua field baru di-sanitasi sisi kode juga (clamp 0-100 tiap skor, trim + drop blank untuk string/array) — tidak percaya sepenuhnya model ikut instruksi format, sama prinsip dengan `sanitizeHashtags` yang sudah ada.
- **Ditampilkan di dua tempat, proporsional dengan ukurannya** — `ClipCard.tsx` (dashboard Clip Gallery, grid rapat) cuma menampilkan `reason` (line-clamped) dan `topics` (chip kecil) — breakdown 7 metrik penuh terlalu ramai untuk kartu grid. `TimelineEditor.tsx`'s panel detail klip (lebih lega, sudah menampilkan hookText/hashtags yang bisa diedit) menampilkan panel lengkap: reason, breakdown 7 skor, badge intent + topics, dan CTA. **Read-only di kedua tempat** di fase ini — beda dari hookText/hashtags yang bisa diedit manual, field-field Fase 8 murni hasil analisis, tidak ada form edit untuk `scores`/`reason`/`topics`/`keywords`/`intent`/`ctaText`.
- **Dashboard heatmap/distribusi/peringkat lintas-klip belum dibangun** — scope fase ini sengaja dibatasi ke level "per-klip" (skor + alasan tampil di kartu/panel klip itu sendiri), bukan visualisasi baru lintas seluruh video (heatmap timeline, distribusi topik/emosi across semua klip, dsb) yang diminta lebih luas — itu ditunda ke fase dashboard terpisah kalau dibutuhkan, supaya fase ini tetap fokus dan bisa diverifikasi end-to-end dulu.
- **Diverifikasi end-to-end dengan audio ucapan sungguhan** (bukan cuma unit test dengan transcript buatan) — video test dibuat dengan Windows `System.Speech` TTS (bukan nada sinus/silent) diproses lewat pipeline penuh (Groq Whisper -> detect-clips -> render-clip) sungguhan; hasilnya membuktikan LLM benar-benar mengekstrak topik/keyword yang relevan dan menangkap CTA ("follow this channel for part 2") yang sengaja disisipkan di skrip ucapan test.

## Smart Trim & Cleanup (Fase 9 pasca-MVP)

Tiga peningkatan render deterministik (bukan berbasis LLM/ML) untuk klip yang lebih rapi: **Smart Start/End** (snap batas klip ke kata sungguhan, `detect-clips.worker.ts`) dan **Silence Removal + Filler Removal** (buang jeda diam panjang & kata pengisi um/uh, `render-clip.worker.ts` + `apps/worker/src/cutlist.ts`).

- **Smart Start/End: snap ke word-boundary Whisper, bukan detik mentah dari LLM** — LLM memilih startTime/endTime dari teks segmen yang diberi timestamp dibulatkan 0.1 detik di prompt, bukan dari audio itu sendiri, jadi angkanya bisa jatuh di tengah kata. `snapToWordBoundaries()` (`detect-clips.worker.ts`) menggeser tiap batas ke kata terdekat: kalau batas jatuh di tengah kata, digeser ke ujung kata itu (klip tidak pernah mulai/berhenti di tengah kata); kalau jatuh di jeda diam, digeser ke kata berikutnya/sebelumnya (bonus: memangkas jeda diam di ujung klip secara gratis). Tanpa data word-level (video lama sebelum Fase 3) atau hasil snap yang tidak valid, dikembalikan ke detik asli tanpa error.
- **Silence Removal + Filler Removal: satu mekanisme cutlist yang sama** (`apps/worker/src/cutlist.ts`) — gap antar-kata lebih dari 0.7 detik jadi kandidat cut (disisakan padding 0.15 detik di tiap ujung supaya tidak jadi jump-cut yang terlalu kasar), dan kata keluarga um/uh (`um`, `umm`, `uh`, `uhh`, `erm`, `hmm`) langsung jadi cut penuh tanpa padding. Kedua jenis cut digabung lewat `mergeCutRanges()` sebelum dipakai.
- **Filler word list SENGAJA sempit** — cuma keluarga um/uh, BUKAN "like"/"so"/"actually"/"basically"/dst yang sering jadi kata sungguhan dalam kalimat ("I like this", "so then what happened"). Heuristik daftar-kata tidak bisa membedakan konteks, jadi dipilih daftar yang nyaris selalu berarti disfluency di kondisi apa pun — lebih sedikit filler yang tertangkap, tapi tidak pernah merusak kalimat. Kejujuran skala yang sama seperti Fase 8's ClipScores.
- **Render dua pass, bukan satu filtergraph rumit dengan remap waktu** — pass pertama (`renderClip()`, tidak berubah sama sekali dari sebelum fase ini) menghasilkan klip ter-crop + caption burn-in seperti biasa di timeline klip ASLI (belum dipotong). Pass kedua (`trimCutRanges()`, `apps/worker/src/ffmpeg.ts`) menjalankan filter `select`/`aselect` (`not(between(t,a,b)+...)` lalu `setpts=N/FRAME_RATE/TB`/`asetpts=N/SR/TB`) di atas HASIL RENDER pass pertama untuk membuang rentang cut. Dipilih dibanding melipat trim+concat ke dalam satu filtergraph `renderClip()` karena caption (ASS) dan crop path (face-tracking sendcmd) burn-in di pass pertama sudah dalam koordinat clip-relative yang SAMA dengan timeline cut — jadi menghapus suatu rentang otomatis ikut menghapus piksel caption/crop di rentang itu juga, tanpa perlu logic remap waktu terpisah untuk caption atau crop path sama sekali. Trade-off yang diterima: satu decode+encode ekstra saat ada cut (dilewati sepenuhnya kalau `cuts.length === 0`, jadi klip tanpa jeda/filler panjang render persis seperti sebelum fase ini, tanpa biaya tambahan).
- **Cutlist dalam koordinat clip-relative (0 = awal klip)**, konsisten dengan konvensi yang sudah ada (`FaceSample.t`, `buildAss`'s internal shift) — `trimCutRanges()` dipanggil pada waktu clip-relative yang SAMA dengan timeline pass-pertama, bukan waktu absolut sumber video.
- **`computeSilenceCuts([], clipDuration)` mengembalikan `[]`, bukan satu cut yang mencakup seluruh klip** — bug nyata yang kejadian saat implementasi: tanpa word-level data sama sekali (video lama, atau segmen tanpa `.words`), logic "trailing gap" awalnya jatuh tembus dan menganggap SELURUH klip sebagai satu jeda diam raksasa. Ketahuan dari test suite yang sudah ada (banyak assertion `cleanupTempFileMock`/upload berubah tak terduga), bukan dari inspeksi manual — pengingat kenapa test regresi terhadap perilaku "tanpa perubahan" tetap penting walau fitur baru terlihat independen.
- **Presisi dibulatkan ke milidetik (`round3`)** — pengurangan floating-point biasa (`word.start - clipStart`) menghasilkan angka seperti `0.2999999999999998`; dibulatkan di titik cut dihitung supaya ekspresi `select` FFmpeg tidak membawa noise sub-milidetik yang tidak berarti apa pun secara visual.
- **Diverifikasi end-to-end dengan audio ucapan sungguhan berisi jeda & filler yang disengaja** — video test dibuat dengan Windows `System.Speech`'s `PromptBuilder` (`AppendBreak` untuk jeda 3s/0.3s/2.5s eksplisit, teks berisi "um"/"uh" literal), diproses lewat pipeline penuh sungguhan. Hasil: log worker melaporkan "removed 6.5s of silence/filler across 3 cut(s)", dan durasi file hasil render (diverifikasi lewat `ffprobe` terhadap file yang benar-benar diunduh) menyusut dari rentang klip asli 23.1 detik menjadi 16.72 detik — selisih ~6.4 detik, cocok dengan yang dilaporkan (perbedaan kecil wajar karena pembulatan frame di filter `select`).

## Auto Zoom — Punch-In Sinematik pada Kata Penekanan (Fase 11 pasca-MVP)

Jalur pertama dari tiga jalur fitur (Visual/CV, Audio Lanjutan, Estetika) yang diminta user untuk dikerjakan bertahap. Menambah **zoom** (crop yang menyempit sesaat lalu kembali) ke sistem smart-reframe Fase 2 yang sebelumnya cuma **pan** (posisi crop mengikuti wajah, ukuran selalu tetap) — deterministik, tanpa LLM/ML tambahan, dipicu murni oleh kata-kata "penekanan" dari transkrip yang sudah ada.

- **Deteksi kata penekanan pakai pola yang sama dengan caption bold-highlight** (`findEmphasisWords()`, `apps/worker/src/reframe.ts`) — angka/persentase, kata ALL-CAPS, dan frasa berkutip, pola yang sama `KEYWORD_PATTERN` di `subtitles.ts`. Sengaja duplikat regex-nya (bukan diekspor-impor) karena stripping punktuasinya beda: `highlightKeywords()` men-strip tanda kutip juga sebelum test (sehingga cabang frasa-berkutip regex itu efektif tidak pernah kena — kutipnya sudah hilang sebelum dicek), sedangkan `findEmphasisWords()` cuma men-strip tanda baca kalimat (`.,!?;:`) supaya cabang kutip beneran bisa kena. Bug laten yang sama di `subtitles.ts` TIDAK ikut diperbaiki di fase ini (di luar scope, dan efeknya cuma caption yang seharusnya di-bold tapi tidak — tidak merusak apa pun).
- **Amplop zoom attack/hold/release, bukan potongan tiba-tiba** (`zoomEnvelopeAt()`) — per kata penekanan: ramp naik 0->1 selama 0.15 detik berakhir tepat saat kata itu mulai diucap (`ZOOM_ATTACK_SECONDS`), tahan di puncak 0.4 detik (`ZOOM_HOLD_SECONDS`), lalu ramp turun 1->0 selama 0.5 detik (`ZOOM_RELEASE_SECONDS`). Kata-kata penekanan yang amplopnya tumpang tindih digabung lewat `Math.max` (ambil zoom terkuat di titik itu), bukan ditumpuk — supaya dua kata penekanan berdekatan tidak bikin crop menyempit berlebihan.
- **Peta zoom maksimum 30% (`MAX_ZOOM_IN_FRACTION`)** — crop menyempit ke 70% ukuran dasarnya di puncak amplop. Angka dipilih supaya terasa sebagai "push-in" sinematik yang jelas terlihat di frame 9:16 talking-head, tanpa terasa disorientasi.
- **`buildCropPath()` sekarang gabungan dua sinyal independen** — posisi (x/y, dari `FaceSample[]`, logika sama seperti Fase 2) dan ukuran (width/height, dari amplop zoom kata-kata penekanan). Salah satu sinyal saja cukup untuk menghasilkan path yang nyata: klip dengan wajah terlacak tapi tanpa kata penekanan tetap pan (tanpa zoom); klip dengan kata penekanan tapi tanpa wajah terlacak tetap zoom (posisi di tengah frame, bukan `null`). `null` cuma dikembalikan kalau KEDUA sinyal kosong (sebelumnya di Fase 2, `null` cukup kalau tidak ada wajah terdeteksi sama sekali) — pemanggil (`render-clip.worker.ts`) baru jatuh ke static center-crop di kondisi yang lebih sempit ini.
- **Path sekarang membentang seluruh `clipDurationSeconds`, bukan cuma sampai sample wajah terakhir** — sebelumnya (Fase 2) path berhenti di `lastSampleT` (biasanya ~1 detik sebelum akhir klip, karena `FACE_SAMPLE_INTERVAL_SECONDS = 1`), meninggalkan ekor klip tanpa titik path eksplisit (`interpolateAt()` kebetulan sudah extrapolate flat, jadi tidak ada bug nyata sebelumnya — tapi juga tidak ada titik path untuk zoom di ekor klip itu kalau ada kata penekanan di sana). Diperbaiki sekaligus sebagai bagian dari fase ini karena zoom butuh cakupan penuh.
- **Dimensi output final (`outputWidth`/`outputHeight`) dipisah dari dimensi crop instan (`width`/`height`/`x`/`y`)** (`ReframeOptions`, `apps/worker/src/ffmpeg.ts`) — crop filter sekarang bisa punya `w`/`h` yang berubah-ubah lewat waktu (menyempit saat zoom), jadi filter `scale=${outputWidth}:${outputHeight}` ditambah SETELAH crop di filtergraph `renderClip()` supaya resolusi frame yang di-encode tetap konstan sepanjang klip — `scale` men-sample ulang apa pun ukuran yang dihasilkan crop di tiap frame balik ke ukuran final, sehingga efeknya terasa seperti zoom kamera, bukan resolusi video yang berubah-ubah.
- **Perbaikan bug laten sekaligus: caption sekarang di-size dari `outputWidth`/`outputHeight`, bukan `width`/`height`** — `render-clip.worker.ts`'s panggilan `buildAss()` sebelumnya pakai `reframe.width`/`reframe.height` (dimensi crop di t=0), yang di Fase 2 SELALU sama dengan dimensi crop dasar (karena crop tidak pernah berubah ukuran). Begitu crop bisa menyempit (zoom di fase ini) dan kebetulan ada kata penekanan yang mulai tepat di t=0, caption akan ter-size untuk frame yang lebih kecil dari frame final — bug yang sebelumnya laten (tidak pernah kena karena Fase 2 tidak pernah mengubah ukuran crop) baru benar-benar bisa muncul begitu zoom ada. Diperbaiki sebagai bagian dari refactor ini, bukan fitur terpisah.
- **`buildSendCmdScript()` sekarang set 4 parameter per baris** (`x`, `y`, `w`, `h`), bukan cuma `x`/`y` — diverifikasi langsung terhadap binary `ffmpeg.exe` sungguhan (bukan cuma lewat mock test) bahwa filter `crop`'s `w`/`h` bisa di-set lewat `sendcmd` di runtime, sama seperti `x`/`y` yang sudah dipakai sejak Fase 2. Dites dengan command file 5-baris yang menyempitkan crop dari 202x360 ke 141x252 lalu kembali, output di-`ffprobe` (dimensi tetap konstan 202x360 sepanjang klip) dan dua frame diekstrak (t=1 "rest" vs t=4.5 "peak") — frame rest menampilkan 4 color bar penuh, frame peak menampilkan cuma 2 (magenta+biru) di-scale mengisi frame yang sama, mengonfirmasi punch-in beneran terjadi di piksel, bukan cuma parameter yang berubah di command file.

Jalur kedua (Audio/Advanced Analysis) dari tiga jalur fitur yang diminta user. Menambah label "siapa yang bicara" per segmen transkrip lewat `pyannote.audio` (Python, model gated di Hugging Face) — krusial untuk video wawancara/panel diskusi seperti yang diminta user secara eksplisit.

- **Skema `TranscriptSegment.speaker String?` sudah ada dari MVP awal, belum pernah diisi** — field ini (juga `TranscriptSegment.speaker?: string` di `packages/shared`, dan pass-through di `transcript-segment.util.ts`) sudah dipersiapkan sejak schema awal tapi tidak ada producer yang mengisinya sampai fase ini. Pemisahan pembicara butuh konteks penuh untuk konsistensi antar-giliran bicara, dan cuma perlu dijalankan sekali terlepas berapa banyak chunk Whisper (`planTranscriptionChunks`) transkrip videonya dipecah. Memuat "-3.1" di versi library ini berarti mencampur segmentation/embedding "-3.1" dengan default PLDA/clustering "community-1" tanpa sengaja — memuat "community-1" langsung menghindari percampuran itu SEKALIGUS cuma perlu accept terms SATU model gated (community-1 membundel segmentation+embedding+plda sendiri dalam satu repo, bukan "-3.1" + dependensi terpisah `pyannote/segmentation-3.0`).

Diverifikasi langsung lewat run sungguhan (bukan cuma dari dokumentasi pyannote): tanpa token sama sekali menghasilkan `RuntimeError: HUGGINGFACE_TOKEN is not set` dengan exit code 1 yang jelas (bukan hang/silent fail); dengan token tapi gated terms belum di-accept menghasilkan `GatedRepoError` (403) yang juga jelas, exit code 1 yang sama.

- **`assignSpeakerLabels()`** (`apps/worker/src/diarization.ts`) memetakan tiap `TranscriptSegment` Whisper ke label pyannote dengan overlap terbesar (bukan per-word, karena segmen Whisper jarang memotong tepat di titik ganti pembicara) — segmen tanpa overlap sama sekali (diarization gagal/`turns=[]`, atau ada gap cakupan) dibiarkan `undefined`, bukan dipaksa ke pembicara terdekat.

- **UI minimal: warna per pembicara di strip transkrip Timeline Editor** (`apps/web/components/TimelineEditor.tsx`) — palet 5 warna tetap (`SPEAKER_COLORS`), index diambil langsung dari huruf label ("Speaker A" -> index 0) karena labelnya sendiri sudah deterministik urut kemunculan, tidak perlu hashing. Video tanpa data speaker (diarization belum jalan/gagal) tetap tampil seperti sebelum fase ini (warna signal-cyan tunggal) - tidak ada perubahan visual untuk video lama.

- **`pipeline(...)` di pyannote.audio 4.x mengembalikan `DiarizeOutput`, bukan `Annotation` langsung** — ditemukan lewat `AttributeError: 'DiarizeOutput' object has no attribute 'itertracks'` di run sungguhan. `Annotation` klasik (yang punya `.itertracks(yield_label=True)`) ada di `output.speaker_diarization`.

## Vocal Emotion Detection (Fase 13 pasca-MVP)

Bagian kedua dari Jalur Audio (Advanced Analysis). Menambah label emosi vokal (dari nada suara, bukan dari kata-katanya) per segmen transkrip — sinyal pelengkap yang jujur diakui sebagai heuristik kasar, bukan skor yang bisa diandalkan sendirian.

- **`TranscriptSegment.emotion String?` — kolom baru, BEDA dari `speaker` di Fase 12** — tidak ada field placeholder yang sudah dipersiapkan sebelumnya untuk ini, jadi fase ini BENAR-BENAR butuh migration baru (`20260705163000_add_transcript_segment_emotion`), tidak seperti `speaker` yang tinggal disambungkan ke plumbing lama. `TranscriptSegment.emotion?: string` juga ditambah ke `packages/shared`, dan pass-through di `transcript-segment.util.ts` + `videos.service.ts`'s `findTranscriptOrThrow` mapping — `filterSegmentsForClip()` tetap tidak perlu diubah (pure filter, otomatis meneruskan field baru apa pun).
- **`superb/wav2vec2-base-superb-er` (taksonomi 4-kelas IEMOCAP: neu/hap/ang/sad) lewat `transformers`'s pipeline `"audio-classification"`** — model publik, TIDAK gated di Hugging Face (beda dari Fase 12's pyannote), jadi tidak butuh `HUGGINGFACE_TOKEN` sama sekali untuk fitur ini — satu kelas blocker manual (accept gated terms) yang hilang dibanding Fase 12.
- **Jalan PER-SEGMEN, bukan sekali per video** — beda dari diarization (Fase 12) yang butuh konteks penuh video untuk konsistensi identitas pembicara. Emosi vokal itu sinyal momen-ke-momen (nada bisa berubah kalimat demi kalimat), jadi tiap `TranscriptSegment` Whisper diklasifikasi independen dari audio slice-nya sendiri — tidak ada alasan untuk "konteks lintas-segmen" di sini.
- **Reuse audio full-track yang SAMA dengan diarization** (`diarizeAudioPath` dari Fase 12) — tidak ada ekstraksi ffmpeg baru sama sekali untuk fase ini. `detect_vocal_emotion.py` menerima path audio itu + satu file JSON berisi daftar rentang segmen (`{start, end}`, ditulis oleh `apps/worker/src/vocalEmotion.ts` ke scratch file sementara miliknya sendiri — beda dari `reframe.ts`'s sendcmd path yang lifetime-nya lintas-modul, file segmen ini murni detail implementasi internal `detectVocalEmotions()`, ditulis dan dibersihkan di fungsi yang sama), lalu men-slice array audio yang sudah didekode SEKALI di memori untuk tiap segmen — model dimuat SEKALI (bukan per segmen) untuk efisiensi.

Video yang diproses aplikasi ini (talking-head asli, wawancara asli) adalah distribusi data yang jauh berbeda dari yang dilatihkan modelnya — hasil klasifikasi HARUS diperlakukan sebagai sinyal pelengkap yang berisik (noisy), sama seperti face detection (Fase 2) dan diarization (Fase 12): berguna kalau benar, aman diabaikan kalau salah, tidak pernah jadi satu-satunya dasar keputusan. TIDAK dipakai sebagai input skor virality (Fase 8's `ClipScores`) di fase ini — murni informasional/tampilan, konsisten dengan bagaimana `speaker` juga baru sebatas tampilan di Fase 12.

Karena baik `diarize_speakers.py` (Fase 12) maupun `detect_vocal_emotion.py` SELALU mendekode audio sendiri lewat `soundfile` dan tidak pernah menyerahkan path file mentah atau objek `torchcodec` ke pyannote/transformers, `torchcodec` (yang cuma ter-install sebagai dependency transitif `torchaudio`, DLL native-nya memang tidak pernah berhasil di-load di setup Windows/Python 3.14 ini) benar-benar tidak dibutuhkan sama sekali — di-uninstall sepenuhnya (`pip uninstall torchcodec`), dan diverifikasi ulang lewat run sungguhan bahwa KEDUA fase (diarization Fase 12 + vocal emotion Fase 13 ini) tetap jalan normal setelahnya. Perbaikan akar masalah, bukan workaround di sekitar gejala.

## Smart Transitions — Dip-to-Black di Titik Potong (Fase 14 pasca-MVP)

Bagian pertama dari Jalur Estetika (Smart Post-Processing). Melembutkan jump-cut yang sebelumnya keras di titik silence/filler removal (Fase 9) dengan transisi dip-to-black singkat — bagian kedua jalur ini (Auto B-roll) menyusul.

- **Scope disempit ke VIDEO-ONLY setelah verifikasi langsung terhadap ffmpeg sungguhan menemukan dua masalah nyata** — bukan keputusan desain awal, murni hasil investigasi: (1) ffmpeg build yang dipakai proyek ini (`8.1.2-full_build` dari gyan.dev) punya bug nyata: chaining DUA instance filter `fade` dalam satu `-vf`/`-filter_complex` membuat SELURUH output jadi hitam, bukan cuma di jendela waktu yang dimaksud — direproduksi berkali-kali (dekat, jauh, `-vf` vs `-filter_complex` dengan label eksplisit, argumen berbasis waktu vs nomor frame), bukan salah syntax. Satu junction dip (fade-out lalu fade-in) SUDAH butuh dua instance `fade`, jadi pendekatan `fade`/`afade` biasa tidak bisa dipakai sama sekali. (2) Alternatif untuk audio (`volume=eval=frame:volume=<ekspresi kondisional>`) diuji langsung dan ternyata tidak reliable: ekspresi LINEAR yang mereferensikan `t` (misal ramp) bekerja benar, tapi ekspresi KONDISIONAL untuk jendela sempit (`if(between(t,a,b),...)` atau `if(lt(abs(t-x),f),...)`) cuma memicu sebagian/tidak konsisten — diukur langsung lewat `volumedetect` terhadap output sungguhan, jendela yang ditarget cuma menunjukkan beberapa dB perubahan, bukan penurunan besar yang diharapkan (kemungkinan besar granularitas frame internal filter audio ini terlalu kasar untuk jendela sub-detik). Daripada mengirim dip audio yang cuma sebagian/tidak konsisten, audio tetap hard-cut apa adanya (tidak berubah dari Fase 9) — titik potongnya sendiri memang silence/filler word yang sudah senyap, jadi jauh lebih tidak mengganggu dibanding jump-cut visual yang fase ini perbaiki.
- **Fix: SATU instance filter `eq` dengan ekspresi brightness time-varying** (`eq=eval=frame:brightness=<expr>`), bukan filter `fade` sama sekali — menghindari bug chaining-dua-fade di atas sepenuhnya. `<expr>` membentuk dip segitiga (0 jauh dari titik potong manapun, -1 tepat di titiknya, linear di antaranya), digabung lintas beberapa titik potong lewat `min()` bertingkat (dip paling gelap yang menang di titik waktu manapun) — SATU instance filter menangani berapa pun jumlah titik potong dalam satu klip, bukan satu filter per titik potong.
- **`computeCutJunctionTimestamps()`** (`apps/worker/src/cutlist.ts`) — fungsi pure baru yang menghitung di mana tiap titik potong (yang sudah di-merge/sort `mergeCutRanges`) jatuh di timeline OUTPUT (setelah `setpts`/`asetpts` memampatkan titik potong sebelumnya keluar dari timeline) — titik pas di mana konten yang tadinya dipisah oleh rentang yang dihapus menjadi sambungan keras. Titik potong pertama sendiri belum tergeser (tidak ada yang dihapus sebelum dia); titik potong berikutnya digeser mundur sebesar total durasi semua titik potong SEBELUMNYA.
- **Titik potong yang terlalu dekat dengan awal/akhir klip di-skip**, bukan tetap di-dip — tidak ada "sebelum"/"sesudah" yang nyata untuk di-dip kalau titik itu ada persis di ujung.
- **Ini transisi "dip" (gaya editing yang sungguhan, biasa dipakai untuk jeda ritme antar kalimat), BUKAN crossfade/dissolve sungguhan** antara konten di kedua sisi — crossfade sungguhan butuh `trimCutRanges`'s pendekatan select-based satu-pass diganti total dengan rantai trim-segmen + `xfade`, perubahan yang jauh lebih besar dan berisiko terhadap fitur yang sudah production (Fase 9) daripada yang dijamin scope fase ini.
- **Verifikasi**: 178 test worker (termasuk test baru `computeCutJunctionTimestamps` di `cutlist.spec.ts` + `trimCutRanges` yang diperbarui di `ffmpeg.spec.ts`) + typecheck + lint semua hijau. **Diverifikasi end-to-end sungguhan** lewat video sintetis berwarna (`smptebars` ffmpeg source, bukan clip TTS yang video-nya cuma hitam polos) dijalankan lewat filter chain PERSIS yang dipakai kode produksi: frame sebelum titik potong menunjukkan warna normal, frame di titik potong menunjukkan warna teredam jelas (dip), frame setelah pulih penuh ke warna asli — bug dua-fade dan fix eq-filter-nya juga masing-masing direproduksi/diverifikasi terpisah sebelum diintegrasikan.

## Auto B-roll (Fase 15 pasca-MVP)

Bagian kedua Jalur Estetika, melengkapi Fase 14 - dengan ini Jalur Estetika (Auto B-roll + Smart Transitions) dan Jalur Audio (Fase 12-13) selesai sepenuhnya; Jalur Visual masih punya sisa (deteksi gestur, eye contact - lihat Roadmap di bawah), Auto Zoom (Fase 11) sendiri sudah selesai lebih dulu. Menyisipkan cutaway stock footage singkat (~2.5 detik, crossfade masuk/keluar) di momen klip yang menyebut kata kunci konkret dari Fase 8.

- **`Pexels` (bukan Pixabay), API PUBLIK, TIDAK gated** — beda dari Fase 12's pyannote, tidak ada langkah "accept model terms" sama sekali. Gratis, tanpa kartu kredit, key langsung muncul di dashboard setelah signup di https://www.pexels.com/api/.
- **Momen dicari dari `Clip.keywords` yang sudah ada sejak Fase 8**, tidak ada panggilan LLM baru — `findBRollMoments()` (`apps/worker/src/broll.ts`) mencari kemunculan PERTAMA tiap keyword di transkrip kata-per-kata klip (cocok untuk keyword satu kata maupun frasa, lewat pencocokan substring pada teks yang digabung, bukan window n-gram), ambil maksimal 2 momen, skip yang terlalu berdekatan (<3.5 detik) atau tidak cukup sisa durasi klip untuk cutaway penuh.
- **`RenderClipJobData` diperluas dengan field `keywords: string[]`** — job payload sebelumnya tidak membawa keywords sama sekali (LLM output Fase 8 cuma tersimpan di kolom `Clip.keywords`, dibaca API buat ditampilkan, tidak pernah diteruskan ke render-clip job). Tiga titik enqueue (`detect-clips.worker.ts`, `ClipsService.render`, `VideosService.retry`) semua diupdate meneruskan field ini.
- **Dua-pass ffmpeg per cutaway, BUKAN satu filter chain dengan fade-in DAN fade-out** — alasan yang SAMA persis dengan Fase 14's dip-to-black: chaining dua instance filter `fade` (satu untuk masuk, satu untuk keluar) terverifikasi membuat SELURUH output hitam di build ffmpeg ini. `trimAndFadeInBRoll()` (pass 1: trim+scale+crop ke ukuran output klip+fade alpha masuk) lalu `fadeOutBRoll()` (pass 2: fade alpha keluar di atas output pass 1) - masing-masing cuma SATU instance filter `fade`.
- **Intermediate file pakai codec `qtrle` (QuickTime Animation) dalam container `.mov`, BUKAN libx264/mp4** — H.264 dalam mp4 standar TIDAK BISA membawa channel alpha sama sekali. `qtrle` lossless dan alpha-capable; ukuran file jauh lebih besar dari H.264 tapi tidak masalah karena file ini cuma dibaca balik oleh `renderClip()`'s `-filter_complex`, tidak pernah jadi output akhir.
- **Ditemukan & dibuang lewat verifikasi langsung**: `colorchannelmixer` (kandidat awal untuk drive alpha lewat `sendcmd`, mekanisme yang sama dengan crop path Fase 2/11) TERNYATA tidak mengimplementasikan command handler sama sekali meski parameternya ditandai "runtime configurable" di opsi filter — ffmpeg sungguhan melaporkan `Command reply: ret:Function not implemented`. Solusi akhir (2-pass qtrle di atas) ditemukan lewat trial nyata, bukan asumsi dari dokumentasi.
- **`renderClip()` (`apps/worker/src/ffmpeg.ts`) beralih dari `-vf` ke `-filter_complex` HANYA kalau ada B-roll** — `-vf`'s shorthand cuma bisa satu input implisit; begitu ada cutaway, tiap satu butuh `-i` tambahan + label bracket eksplisit (`[N:v]setpts=PTS-STARTPTS+START/TB[brollN]` lalu `overlay=enable='between(t,START,END)'`). Jalur `-vf` lama (tanpa B-roll) tidak diubah sama sekali - risiko regresi nol untuk klip tanpa cutaway.
- **Urutan filter: crop/zoom (Fase 2/11) -> overlay B-roll -> subtitle** — caption tetap terlihat di atas cutaway (aksesibilitas: penonton tetap bisa baca apa yang diucapkan walau visual berpindah sebentar ke B-roll), overlay terjadi SETELAH crop supaya cutaway pas mengisi frame 9:16 yang sudah benar, bukan frame lebar asli.
- **Kegagalan per-momen, bukan per-klip** — satu momen gagal (Pexels tidak ada hasil, network error, `PEXELS_API_KEY` belum di-set) cuma di-skip momen itu, klip tetap render tanpa cutaway di titik itu, bukan menggagalkan seluruh job. `PEXELS_API_KEY` sengaja TIDAK masuk `REQUIRED_ENV_VARS` (`env.ts`).
- **Verifikasi**: 203 test worker (termasuk `broll.spec.ts` baru + describe block baru `renderClip`/`trimAndFadeInBRoll`/`fadeOutBRoll` di `ffmpeg.spec.ts` + `render-clip.worker.spec.ts`) + typecheck + lint semua hijau di `apps/worker`/`apps/api`/`apps/shared`. **Diverifikasi end-to-end SUNGGUHAN dengan Pexels API key asli**: pencarian sungguhan untuk keyword "sunset" mengembalikan video sungguhan, di-download, diproses lewat kedua pass ffmpeg sungguhan, dikomposit ke video utama sintetis lewat `-filter_complex` PERSIS yang dipakai produksi — frame sebelum cutaway menunjukkan video utama, frame di tengah cutaway menunjukkan footage sunset sungguhan mengisi penuh frame 9:16, frame setelah cutaway kembali ke video utama tanpa artefak.
- **Belum ditangani**: Docker image `apps/worker` belum diverifikasi punya `qtrle` encoder (harusnya sudah termasuk default di build ffmpeg manapun yang sudah dipakai proyek ini, tapi belum dicek eksplisit terhadap image container), dan seperti Fase 12/13, belum ada langkah setup Pexels khusus di Dockerfile (tidak perlu — cuma env var, tidak ada dependency sistem baru).

## Multi-Provider Stock Assets — Adapter Pattern (Fase 16 pasca-MVP)

Permintaan eksplisit user untuk merombak Fase 15's integrasi Pexels tunggal menjadi sistem 3-provider (Pexels, Pixabay, Unsplash) di belakang Adapter pattern yang bersih/decoupled, supaya `ffmpeg.ts`/render pipeline tidak pernah perlu tahu dari provider mana sebuah asset berasal.

- **`StockAsset`** (`apps/worker/src/assets/types.ts`) — bentuk data yang dinormalisasi: `{id, url, thumbnail, sourceName, resolution: {width,height}, type: 'video'|'image'}`. SEMUA konsumen (`StockAssetService`, `render-clip.worker.ts`, `ffmpeg.ts`) cuma pernah melihat bentuk ini, tidak pernah respons mentah provider manapun — itu inti dari Adapter pattern di sini.
- **`AssetProvider`** — satu interface yang diimplementasikan tiap adapter (`PexelsAdapter`, `PixabayAdapter`, `UnsplashAdapter`, semua di `apps/worker/src/assets/`), method `search(keyword): Promise<StockAsset | null>`. `null` untuk "tidak ada yang bisa dipakai" (API key belum di-set, tidak ada hasil, bentuk respons tak terduga) — BUKAN error; error jaringan/API sungguhan (rate limit, down) dibiarkan throw supaya `StockAssetService`'s try/catch per-provider bisa menangkapnya dan lanjut ke provider berikutnya, beda dari "tidak ada hasil" yang memang mestinya diam-diam dilewati.
- **`StockAssetService`** (orchestrator) — `searchAssets(keyword)` mencoba tiap provider, tier demi tier, berhenti di hasil pertama yang bisa dipakai:
  - Tier 1 (video): `PexelsAdapter`, `PixabayAdapter` — diprioritaskan karena B-roll berupa video bergerak terasa lebih baik daripada foto diam.
  - Tier 2 (foto): `UnsplashAdapter` — fallback kalau tidak ada satupun provider video yang punya hasil untuk keyword itu.
  - Satu provider throw (down/rate-limit/respons aneh) ditangkap DI SINI, dicatat via `console.warn`, lanjut ke provider berikutnya — pipeline render TIDAK PERNAH crash gara-gara satu provider stock asset bermasalah.
- **`StockAssetCache`** — cache in-memory sederhana (TTL 24 jam), key berdasarkan keyword pencarian. Hasil NEGATIF (`null`, tidak ada provider manapun yang punya sesuatu) JUGA di-cache, bukan cuma hasil positif — ini yang benar-benar menghemat quota untuk keyword yang memang tidak ada footage-nya (tanpa ini, tiap klip yang menyebut keyword itu akan query ulang ke 3 provider lagi). In-memory (bukan DB) - reset saat worker restart, trade-off yang diterima karena cache miss cuma berarti search ulang, bukan kegagalan apa pun.
- **`httpClient.ts`** — wrapper `fetch` bersama (`fetchJson<T>()`) dipakai ketiga adapter, bukan axios (monorepo ini tidak punya axios sama sekali, dan native `fetch` Node 20+ sudah dipakai di tempat lain - lihat broll.ts). Timeout eksplisit (10 detik default) + `HttpRequestError` (dengan `status`) untuk respons non-2xx, jadi rate-limit/outage sungguhan gampang dibedakan dari "tidak ada hasil".
- **Mapping tiap provider (bagian paling gampang salah, jadi didokumentasikan detail per method)**:
  - `PexelsAdapter`: Video Search API, pilih `video_files` terkecil yang masih >= 480px lebar (fallback ke terbesar) — file lebih besar cuma buang-buang bandwidth untuk cutaway ~2.5 detik yang di-scale/crop lagi.
  - `PixabayAdapter`: Video API, `per_page=3` (minimum yang diterima API, cuma hit pertama yang dipakai). Field thumbnail TIDAK ADA langsung di respons video Pixabay — dikonstruksi dari `picture_id` lewat pola resmi terdokumentasi Pixabay sendiri (`https://i.vimeocdn.com/video/{picture_id}_{size}.jpg`), bukan trik reverse-engineered.
  - `UnsplashAdapter`: Search Photos API, selalu `type: 'image'` (Unsplash tidak punya video). **Kepatuhan API yang sungguhan diimplementasikan, bukan cuma di-skip**: Unsplash's API Guidelines mewajibkan ping ke endpoint `photo.links.download_location` setiap kali foto BENAR-BENAR dipakai (bukan cuma tampil di hasil pencarian) — diimplementasikan sebagai fire-and-forget async (sesuai anjuran Unsplash sendiri: "trigger asynchronously") yang gagalnya di-swallow (tidak pernah menggagalkan pencarian itu sendiri).
- **`ffmpeg.ts`'s `trimAndFadeInBRoll()` diperluas dengan parameter `assetType: 'video' | 'image'`** — video diputar+dipotong seperti biasa; image di-loop lewat `-f image2 -loop 1` (eksplisit, tidak mengandalkan ekstensi file yang di-download, karena URL stock API sering tidak punya ekstensi bersih) supaya satu frame diam menghasilkan durasi cutaway yang sama seperti video. Ini PERSIS "prinsip utama" yang diminta user: `ffmpeg.ts` cuma perlu tahu `'video'|'image'`, tidak pernah perlu tahu asetnya dari Pexels/Pixabay/Unsplash.
- **Migrasi dari Fase 15**: `broll.ts`'s `searchPexelsVideo()` dihapus (digantikan `PexelsAdapter` + `StockAssetService`), `downloadBRollVideo()` di-generalisasi jadi `downloadStockAsset()` (logic download-nya sendiri sama persis untuk video maupun foto — cuma `trimAndFadeInBRoll` yang bedakan). `RenderClipJobData`/`render-clip.worker.ts` tidak berubah strukturnya (masih `keywords: string[]` yang sama dari Fase 15), cuma pemanggilan pencarian aset yang beralih dari `searchPexelsVideo(keyword)` langsung ke `stockAssetService.searchAssets(keyword)`.
- **Konfigurasi**: `PEXELS_API_KEY`, `PIXABAY_API_KEY`, `UNSPLASH_ACCESS_KEY` — ketiganya independen opsional (`env.ts`), TIDAK masuk `REQUIRED_ENV_VARS`. Worker tanpa satupun key tetap render normal, cuma tanpa B-roll; worker dengan sebagian key tetap dapat B-roll dari yang ada.
- **Bug nyata ditemukan & diperbaiki lewat verifikasi sungguhan terhadap Pixabay**: rencana awal (berdasarkan riset dokumentasi + tipe komunitas, bukan tebakan asal — lihat riwayat sesi) mengira Pixabay's Video API tidak punya field thumbnail langsung dan perlu dikonstruksi dari field `picture_id` lewat pola URL vimeocdn. Setelah `PIXABAY_API_KEY` sungguhan tersedia dan dipanggil langsung ke API asli, respons NYATA sama sekali tidak punya field `picture_id` di level manapun — melainkan tiap varian ukuran video (`large`/`medium`/`small`/`tiny`) punya field `thumbnail` LANGSUNG sendiri-sendiri. Riset/dokumentasi yang dipakai sebelumnya rupanya usang untuk versi API Pixabay saat ini. Diperbaiki dengan menghapus `buildThumbnailUrl`/`THUMBNAIL_SIZES` sepenuhnya dan langsung memakai `chosen.thumbnail` dari varian yang dipilih — jauh lebih sederhana DAN benar. Pengingat kuat lain kenapa "diverifikasi lewat riset dokumentasi" tetap harus dianggap sementara sampai benar-benar dipanggil ke API sungguhan sebelum diklaim selesai.
- **Verifikasi**: 234 test worker (34 test baru: `httpClient.spec.ts`, `pexelsAdapter.spec.ts`, `pixabayAdapter.spec.ts`, `unsplashAdapter.spec.ts`, `stockAssetCache.spec.ts`, `stockAssetService.spec.ts`, plus update `broll.spec.ts`/`ffmpeg.spec.ts`/`render-clip.worker.spec.ts`) + typecheck + lint semua hijau. **SEMUA TIGA provider + `StockAssetService` diverifikasi end-to-end SUNGGUHAN** lewat `tsx` menjalankan kode produksi asli (bukan mock) dengan ketiga API key sungguhan: `PexelsAdapter`, `PixabayAdapter` (setelah fix thumbnail di atas), dan `UnsplashAdapter` (termasuk ping `download_location` compliance-nya, tidak error) semua mengembalikan `StockAsset` valid dan lengkap dari API asli masing-masing; `StockAssetService.searchAssets()` sungguhan mengembalikan hasil Tier 1 (Pexels) tanpa perlu fallback (semua provider terkonfigurasi dan sehat). Dijalankan ulang tanpa API key apa pun mengembalikan `null` dengan bersih, tanpa crash — mengonfirmasi tiered fallback + graceful-degradation-nya sungguhan bekerja, tidak cuma di unit test yang di-mock.

## Normalisasi B-roll — FPS & Color Space (Fase 17 pasca-MVP)

Permintaan lanjutan user: aset stock dari 3 provider (Fase 16) datang dengan resolusi/framerate/color space yang beda-beda — perlu dinormalisasi ke standar yang konsisten supaya tidak ada lompatan kontras/framerate yang mencolok saat cutaway B-roll muncul di tengah klip.

- **Resolusi**: SUDAH benar sejak Fase 15/16 (`scale=...,crop=...` ke `outputWidth`/`outputHeight` KLIP ITU SENDIRI, bukan konstanta "1080p" hardcoded — hardcode akan salah kalau output klip yang sebenarnya beda ukuran). Tidak ada perubahan di bagian ini.
- **Framerate**: ditambah `fps=BROLL_TARGET_FPS` (30) ke filter chain `trimAndFadeInBRoll` (`apps/worker/src/ffmpeg.ts`) — stock footage datang dari ~24fps sampai 60fps (Pexels sering 60fps), dinormalisasi ke satu angka pasti sebelum masuk tahap composite, bukan mengandalkan `overlay`/framesync ffmpeg mendamaikan framerate berbeda-beda secara implisit saat runtime.
- **Color space**: ditambah `colorspace=iall=bt709:all=bt709:range=tv` — memaksa SEMUA aset (apa pun matrix/primaries/range yang dideklarasikan sumbernya, atau TIDAK dideklarasikan sama sekali) ke satu standar bt709 + range "tv" (limited, 16-235) yang dikenal pasti, sebelum di-composite ke klip utama.
- **Bug nyata ditemukan & diperbaiki lewat verifikasi langsung**: percobaan pertama cuma pakai `all=bt709:range=tv` (tanpa `iall=`) — begitu diuji terhadap footage dengan metadata warna tidak lengkap/"unknown" (kasus umum untuk video yang beredar di web), filter `colorspace` LANGSUNG error `Unsupported input primaries 2 (unknown)` dan menggagalkan seluruh pass. Diperbaiki dengan menambah `iall=bt709` (asumsi INPUT eksplisit) — filter tidak lagi bergantung pada metadata sumber yang mungkin tidak lengkap, selalu punya titik awal konversi yang pasti.
- **Temuan tambahan (bukan bug, cuma perlu dipahami)**: pixel format akhir yang benar-benar di-encode `qtrle` adalah `argb` (RGB+alpha), BUKAN `yuva420p` yang diminta filter chain — `qtrle` cuma menerima varian RGB, jadi ffmpeg otomatis mengonversi di titik encode. Ini JUSTRU menguntungkan tujuan "YCbCr vs RGB alignment": karena SEMUA aset (apa pun matrix aslinya) dipaksa ke RGB dengan primaries/transfer bt709 yang SAMA lewat filter `colorspace` sebelum encode, konversi YUV->RGB akhirnya sudah otomatis konsisten lintas aset — bukan sesuatu yang perlu ditangani terpisah.
- **Scope**: normalisasi ini cuma berlaku untuk PERSIAPAN aset B-roll (`trimAndFadeInBRoll`), bukan pipeline render klip utama (`renderClip()`) yang tidak diubah — video utama tetap mengandalkan default ffmpeg/libx264 seperti sebelumnya. Untuk sumber HD/portrait modern (kasus dominan aplikasi ini), default itu pada praktiknya sudah bt709 juga, jadi keselarasan tetap terjaga di kasus umum; keselarasan warna end-to-end PENUH (memaksa klip utama JUGA ke bt709 eksplisit) di luar scope permintaan ini.
- **Verifikasi**: test `ffmpeg.spec.ts` diperbarui untuk filter string baru, seluruh 234 test worker + typecheck + lint tetap hijau. **Diverifikasi end-to-end SUNGGUHAN** dengan footage sungguhan dari 2 provider berbeda sekaligus (real Pexels video ~29.97fps bt709/tv, dan real Unsplash JPEG statis yang di-loop) — keduanya, meski sumber & tipe asetnya beda total, sama-sama menghasilkan output ternormalisasi PERSIS `360x640 @ 30fps`; frame diekstrak dan diperiksa visual langsung (bukan cuma metadata) untuk keduanya - warna natural, tidak ada pergeseran tint/kontras aneh dari konversi color space.

## Seamless Long-Video Chunking — Overlap + Nominal-Ownership Filter (Fase 18 pasca-MVP)

Permintaan lanjutan user: video panjang (>50 menit) dipecah jadi beberapa chunk Whisper (`MAX_TRANSCRIBE_SECONDS`, `apps/worker/src/workers/transcribe.worker.ts`) karena batas API — sebelumnya tiap chunk diekstrak PERSIS di batas nominalnya, jadi kata yang kebetulan sedang diucapkan tepat di detik ke-3000 (batas chunk) bisa terpotong di tengah kata sebelum sampai ke Whisper sama sekali. Diterima sebagai tradeoff sebelumnya ("clipped/dropped"); permintaan ini minta itu benar-benar diperbaiki, bukan cuma didokumentasikan sebagai keterbatasan.

- **Dua window per chunk, bukan satu**: window NOMINAL (dari `planTranscriptionChunks`, tidak berubah) tetap menentukan chunk mana yang "memiliki" suatu momen absolut begitu semua transkrip digabung. Window EKSTRAKSI baru (`computeChunkExtractionWindow`) melebarkan window nominal itu `CHUNK_OVERLAP_SECONDS` (15 detik) di kedua sisi, di-clamp ke `[0, totalDurationSeconds]` — inilah audio yang BENAR-BENAR diekstrak dan dikirim ke Whisper untuk chunk tersebut.
- **Kenapa 15 detik cukup**: tidak ada kata/frasa yang diucapkan sepanjang itu, jadi Whisper SELALU dapat konteks penuh di sekitar batas 50 menit manapun — tidak pernah ada potongan sample-accurate yang jatuh di tengah kata, siapa pun chunk yang akhirnya "memenangkan" momen itu.
- **Filter ownership deterministik, bukan fuzzy dedup**: karena window ekstraksi kedua chunk tetangga saling tumpang tindih di zona 15 detik itu, KEDUANYA akan mentranskrip audio yang sama persis di zona itu — hasilnya, sebelum digabung ke transkrip final, tiap segment/word disaring: `absStart >= nominalStart && absStart < nominalEnd`. Chunk yang bukan pemilik nominal suatu momen membuang "fringe" duplikatnya sendiri. Ini murni aritmetika timestamp, tidak butuh algoritma pencocokan teks/similarity yang jauh lebih kompleks dan tidak deterministik.
- **Offset berubah dari `chunk.startSeconds` (nominal, lama) ke `window.startSeconds` (ekstraksi aktual, baru)** — timestamp yang dikembalikan Whisper selalu 0-based relatif terhadap audio yang BENAR-BENAR diterimanya, bukan terhadap batas nominal chunk.
- **Clamping otomatis menangani edge case chunk pertama/terakhir** tanpa percabangan eksplisit "apakah ini chunk pertama/terakhir": `Math.max(0, ...)` bikin chunk pertama otomatis tidak punya overlap di depan, `Math.min(totalDurationSeconds, ...)` bikin chunk terakhir otomatis tidak punya overlap di belakang.
- **`singleRequest` (video pendek, satu chunk) sengaja tidak disentuh**: tidak ada window ekstraksi yang dilebarkan, tidak ada filter nominal yang diterapkan — tidak ada apa pun yang perlu di-stitch, dan `chunk.durationSeconds` di kasus itu adalah panjang FULL track, bukan batas per-chunk yang nyata.
- **Verifikasi**: 4 unit test baru untuk `computeChunkExtractionWindow` (widen normal, clamp chunk pertama, clamp chunk terakhir, clamp kedua sisi sekaligus untuk video pendek) + test existing untuk alur multi-chunk ditulis ulang dengan data "ghost" segment/word yang sengaja ditaruh di fringe overlap tiap chunk, untuk membuktikan filter ownership benar-benar membuang duplikat dan bukan cuma kebetulan lolos. 238/238 test worker + typecheck + lint hijau. **Belum diverifikasi end-to-end dengan audio panjang sungguhan (>50 menit) via Whisper API sungguhan** — beda dengan kebanyakan fase lain di sesi ini yang punya verifikasi real end-to-end, generate + transkrip audio 50+ menit sungguhan dianggap tidak praktis untuk sesi ini; ini murni aritmetika timestamp (bukan filter graph ffmpeg yang berisiko punya perilaku mengejutkan), jadi risiko tersembunyi dianggap rendah, tapi ini tetap gap verifikasi yang jujur perlu diakui.

## Keputusan Arsitektur

- **BullMQ dipakai untuk semua kerja berat/async** (transcribe, detect-clips, render-clip). API layer tidak pernah menjalankan Whisper atau FFmpeg secara sinkron di request-response cycle.
- **Worker dipisah dari API** supaya FFmpeg cluster dan proses ASR yang CPU/GPU-intensive bisa di-scale terpisah dari layer API yang menangani traffic HTTP.
- **PostgreSQL sebagai source of truth** untuk status job dan metadata video/klip; Redis hanya untuk antrian (BullMQ) dan cache, bukan penyimpanan permanen.
- **Status video/klip berbentuk state machine linear** (`UPLOADED -> TRANSCRIBED -> CLIPS_DETECTED -> RENDERED`) yang disimpan di PostgreSQL agar frontend bisa polling progres secara konsisten.
- **`Video.transcribeProgress` (0-100, nullable) memberi progres asli di dalam tahap Transcribe** — sebelumnya progress bar `apps/web` cuma punya 4 titik state machine di atas (melompat 0/33/66/100%, flat sepanjang tahap Transcribe walau videonya lama ditranskrip). `transcribe.worker.ts` menulis checkpoint (`reportProgress()`) langsung ke Postgres — bukan lewat BullMQ's `job.updateProgress()` — di titik: reset ke 0 saat job mulai (termasuk retry, supaya tidak menampilkan sisa progres dari percobaan yang gagal), 5 setelah source selesai didownload, lalu per-chunk (10 + porsi dari 80 berdasarkan jumlah chunk yang sudah selesai) — video pendek (kasus umum, satu chunk) cuma dapat satu lompatan ke 90% begitu satu-satunya panggilan Whisper selesai, karena memang tidak ada sinyal progres yang lebih halus dari satu API round-trip. Direset ke `null` begitu status pindah ke `TRANSCRIBED`. Sengaja **tidak** ada animasi/interpolasi buatan — kalau tidak ada checkpoint baru, bar diam di angka terakhir yang benar-benar dicapai.
- **Prisma di `packages/database` sebagai satu-satunya akses ke PostgreSQL**, dipakai baik oleh `apps/api` maupun `apps/worker` (model: `User`, `Video`, `TranscriptSegment`, `Clip`, `SocialAccount` — lihat `packages/database/prisma/schema.prisma`). Transcript segment disimpan per-video (bukan diduplikasi per-klip); transcript sebuah klip didapat dengan query segment dalam rentang `startTime`-`endTime` klip tersebut.
- **Video disimpan di object storage S3-compatible** (`packages/storage`, dipakai bareng oleh `apps/api` dan `apps/worker`), bukan local disk — supaya `apps/api` dan `apps/worker` (proses terpisah, bisa jalan di mesin/container berbeda) tidak perlu berbagi filesystem. `apps/api/src/storage` upload buffer hasil multer langsung ke bucket lewat `uploadObject()`, `Video.sourceUrl` menyimpan **object key** (`videos/<uuid>.<ext>`), bukan path. Konfigurasi lewat env var generik `STORAGE_*` (`STORAGE_ENDPOINT`, `STORAGE_REGION`, `STORAGE_BUCKET`, `STORAGE_ACCESS_KEY_ID`, `STORAGE_SECRET_ACCESS_KEY`, `STORAGE_FORCE_PATH_STYLE`) — bukan `R2_*` — supaya provider-nya bisa diganti (mis. AWS S3) tanpa ubah kode, walau dev saat ini dites terhadap Cloudflare R2 asli. `packages/storage` pakai lazy singleton (`getClient()`, konstruksi `S3Client` ditunda sampai first use) untuk hindari bug class baca `process.env` sebelum `.env` ke-load — sama seperti kasus `QueueModule`/`JwtStrategy`.
- **`S3Client`'s `requestHandler` diberi `connectionTimeout`/`requestTimeout` (+ `throwOnRequestTimeout: true`) eksplisit** — AWS SDK v3's default request timeout adalah 0 (tidak ada batas sama sekali). Bug nyata pernah kejadian: koneksi keep-alive dari connection pool client yang sama basi (berhasil sekali, lalu panggilan berikutnya di client yang sama persis hang selamanya tanpa error apa pun) terhadap R2 — reproducible persis "berhasil sekali, request kedua di client yang sama hang". Tanpa batas, ini bikin upload/preview video hang selamanya (bukan gagal dengan error) baik di `uploadObject()` maupun `getObjectStreamRange()` (dipakai preview timeline editor) karena keduanya berbagi lazy singleton client yang sama. `requestTimeout` diset generous (5 menit) supaya tidak mengganggu upload video besar yang sah di koneksi lambat, cuma membatasi kasus koneksi benar-benar mati.
- **`apps/worker` baca source video langsung dari object storage tanpa nyentuh disk untuk job `transcribe`** — `getObjectStream()` mengembalikan Node `Readable`, di-pipe langsung ke `toFile()` (helper dari OpenAI SDK yang terima `AsyncIterable`) lalu masuk ke Whisper API. Untuk job `render-clip`, FFmpeg **butuh path file lokal asli** (tidak bisa seek/tulis langsung ke object storage), jadi `apps/worker/src/storage.ts` cuma menyediakan **scratch space sementara** di `os.tmpdir()` (`reserveScratchPath()`) — source didownload ke situ, FFmpeg proses di situ, hasil render diupload balik ke bucket (`renders/<clipId>.mp4`), lalu semua file scratch (source/SRT/output) dihapus di blok `finally` terlepas sukses/gagal.
- **Worker meng-update status video sendiri** setelah job selesai (mis. `transcribe` job set status `TRANSCRIBED` setelah berhasil, atau `FAILED` kalau error), bukan lewat callback ke `apps/api`.
- **Worker self-chains job berikutnya dalam pipeline**: `transcribe` job yang sukses langsung enqueue job `detect-clips` sendiri, dan `detect-clips` yang sukses langsung enqueue satu job `render-clip` per kandidat klip (lihat `apps/worker/src/queues.ts`), bukan lewat `apps/api`. Orkestrasi antar-tahap pipeline berada di `apps/worker`, bukan di layer API.
- **FFmpeg dipanggil langsung via `child_process`** (bukan wrapper library seperti `fluent-ffmpeg`) dari `apps/worker/src/ffmpeg.ts`. Butuh binary FFmpeg tersedia (di `PATH`, atau via `FFMPEG_PATH`) di environment tempat `apps/worker` jalan.
- **Render output diupload ke object storage juga** (prefix `renders/`, key `renders/<clipId>.mp4`), konsisten dengan keputusan storage video asli — `apps/worker` satu-satunya penulis object ini.
- **`GET /videos/:id` tidak mengembalikan `Clip.outputUrl` mentah** (object storage key) ke client; di-map jadi `downloadUrl` (`/clips/:id/download` relatif) supaya detail implementasi storage tidak bocor ke API response. `GET /clips/:id/download` di `apps/api` stream file dari bucket lewat `getObjectStream()` (bukan `res.download()` ke path lokal lagi).
- **CORS di `apps/api` di-enable eksplisit dengan `credentials: true`** (`WEB_ORIGIN`, default `http://localhost:3000`) supaya `apps/web` (origin beda, port 3000) bisa manggil `apps/api` (port 3001) dari browser sambil ikut kirim cookie sesi.
- **Auth: email + password + JWT di httpOnly cookie**, bukan token di `localStorage`/`Authorization` header. `AuthModule` (`apps/api/src/auth`) — `POST /auth/register` (bcrypt hash, langsung login), `POST /auth/login`, `POST /auth/logout`, `GET /auth/me`. `JwtStrategy` extract token dari cookie `token`, bukan `Bearer` header. `ownerId` untuk video **selalu** diambil dari user di sesi (`@CurrentUser()`), tidak pernah dari body request — endpoint lama `POST /users` (get-or-create by email tanpa password) sudah dihapus karena itu lubang keamanan (siapa saja bisa "jadi" user manapun cuma dengan tahu emailnya).
- **`GET /videos/:id` dan `GET /clips/:id/download` mengecek kepemilikan** (`video.ownerId`/`clip.video.ownerId` harus sama dengan user di sesi) dan melempar 404 yang sama baik untuk resource yang tidak ada maupun resource milik user lain — supaya endpoint tidak bisa dipakai untuk menebak ID mana yang valid.
- **`apps/web` punya dua route**: `/` (upload video baru + progress live untuk video yang baru saja di-upload) dan `/dashboard` (riwayat semua video milik user, lewat `GET /videos`). Keduanya pakai `useAuth()` hook yang sama (`lib/useAuth.ts`) untuk cek sesi via `GET /auth/me` — bukan `localStorage` — supaya status login selalu konsisten dengan sesi di server.
- **`POST /auth/login` di-rate-limit** (5 percobaan / 60 detik, per IP, via `@nestjs/throttler`). `ThrottlerGuard` **tidak** didaftarkan global — cuma dipasang `@UseGuards(ThrottlerGuard)` langsung di route `login`, jadi endpoint lain (termasuk `/auth/register`) tidak kena efeknya. Config in-memory (bukan Redis-backed) karena `apps/api` masih jalan sebagai satu instance untuk MVP ini.
- **Env var wajib divalidasi saat boot, gagal cepat kalau ada yang kosong** — `apps/api/src/config/env.validation.ts` (dipasang via `ConfigModule.forRoot({ validate })`, jalan sinkron sebelum modul lain di-build) dan `apps/worker/src/env.ts` (dipanggil di `main.ts` tepat setelah `dotenv.config()`, sebelum modul lain yang baca `process.env` di-import). Cuma var tanpa fallback aman yang wajib (`DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`/`OPENAI_API_KEY`, `STORAGE_*`) — var yang sudah punya default masuk akal di kode (`WEB_ORIGIN`, `API_PORT`, `JWT_EXPIRES_IN`, `FFMPEG_PATH`) tidak diwajibkan. Tujuannya supaya `JWT_SECRET` kosong (yang tanpa ini akan diam-diam sign token dengan `undefined`) atau `STORAGE_*` kosong (yang tanpa ini akan gagal jauh di dalam percobaan koneksi S3) langsung ketahuan di boot, bukan nanti pas runtime.
- **`apps/api` pakai `helmet()`** (security response headers: CSP, HSTS, X-Frame-Options, dst — lihat `apps/api/src/main.ts`) dan punya endpoint **`GET /health`** (tidak perlu auth, tidak di-rate-limit — dipanggil load balancer/orchestrator) yang query `SELECT 1` ke Postgres dan balas `503` kalau DB tidak bisa dijangkau.
- **Tiap app punya `Dockerfile` multi-stage sendiri**, di-build dari root repo (workspace deps butuh akses ke `packages/*`). Pakai `pnpm deploy --prod --ignore-scripts` untuk bundel dist + production deps app tersebut jadi direktori yang self-contained tanpa symlink workspace (`--ignore-scripts` karena kalau tidak, postinstall `packages/database` — `prisma generate` — jalan ulang di install `--prod`, padahal `prisma` CLI-nya sendiri devDependency yang sudah kepangkas; generated client dari build step sebelumnya sudah ikut ter-copy apa adanya). Tidak ada `.env` yang di-copy ke image manapun — `envFilePath`/`dotenv.config()` yang dipakai `apps/api`/`apps/worker` cuma no-op kalau file-nya tidak ada, sama seperti perilakunya di CI, dan config sepenuhnya datang dari environment variable asli yang dikasih saat container jalan. `apps/worker`'s image nginstall `ffmpeg` asli lewat `apk` — kalau override `FFMPEG_PATH`, pastikan itu path di *dalam* container, bukan path host (gampang kebawa kalau reuse `.env` dev apa adanya buat container). `apps/web`'s image pakai Next.js `output: 'standalone'` (lihat `next.config.mjs`) plus `NEXT_PUBLIC_API_URL` sebagai build arg (bukan runtime env — `NEXT_PUBLIC_*` di-inline ke bundle client saat `next build`). Migration jalan lewat `packages/database/Dockerfile` terpisah (single-stage, full install biar `prisma` CLI-nya ada) sebagai one-shot job sebelum `apps/api`/`apps/worker` start — lihat `docker-compose.prod.yml`, service `migrate` pakai `condition: service_completed_successfully` supaya api/worker nunggu migration selesai dulu.

## Konvensi Coding

- Bahasa: TypeScript di seluruh monorepo (`apps/web`, `apps/api`, `apps/worker`, `packages/shared`).
- Semua kontrak data (job payload, DTO API, enum status) didefinisikan di `packages/shared` dan diimpor, bukan diduplikasi.
- Job BullMQ dinamai dengan verb-noun (`transcribe`, `detect-clips`, `render-clip`) dan payload/return type-nya didefinisikan di `packages/shared`.
- Perubahan skema PostgreSQL melalui migration (bukan sync otomatis) agar histori skema terlacak.
