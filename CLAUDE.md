# CLAUDE.md

Referensi arsitektur & konvensi untuk **viral-clip-app** â€” platform AI video repurposing (mirip OpusClip) yang mengubah video panjang menjadi klip pendek siap-viral secara otomatis.

## Ringkasan Produk

Alur inti MVP:

```
Upload video -> Transcript (ASR) -> Auto-clip (deteksi momen menarik) -> Caption (burn-in) -> Download
```

## Tech Stack

| Layer | Teknologi |
|---|---|
| Frontend | Next.js + TypeScript |
| Backend API | NestJS |
| Database | PostgreSQL (via Prisma ORM di `packages/database`, dipakai `apps/api` & `apps/worker`) |
| Queue / Cache | Redis + BullMQ |
| Video processing | FFmpeg cluster (worker nodes terpisah) |
| ASR (speech-to-text) | Whisper (OpenAI audio transcription API) |

## Struktur Monorepo

```
apps/
  web/        # Next.js frontend â€” upload UI, editor klip, preview, dashboard
  api/        # NestJS backend â€” REST/GraphQL API, auth, job orchestration
  worker/     # Job consumer â€” ASR (Whisper), auto-clip detection, FFmpeg render, captioning
packages/
  shared/     # Tipe TypeScript, DTO, konstanta, util yang dipakai lintas apps
  database/   # Prisma schema/client Postgres, dipakai apps/api dan apps/worker
  storage/    # Klien object storage S3-compatible (upload/download/delete), dipakai apps/api dan apps/worker
  social/     # OAuth client, enkripsi token, upload klien per-platform (YouTube, TikTok, Instagram), dipakai apps/api dan apps/worker
```

- `apps/web` dan `apps/api` hanya berkomunikasi lewat HTTP API â€” tidak ada import langsung antar keduanya.
- `apps/worker` tidak melayani HTTP; hanya mengonsumsi job dari BullMQ queue yang di-enqueue oleh `apps/api`.
- Tipe yang dibagi antara frontend, backend, dan worker (mis. bentuk payload job, status enum, DTO) didefinisikan sekali di `packages/shared` â€” jangan duplikasi tipe di masing-masing app.

## Alur Pemrosesan Video (MVP)

1. **Upload** â€” `apps/web` upload file ke `apps/api`, video disimpan (object storage), record dibuat di PostgreSQL dengan status `UPLOADED`.
2. **Transcript** â€” `apps/api` enqueue job `transcribe` ke BullMQ. `apps/worker` menjalankan Whisper, hasil transcript (dengan timestamp) disimpan ke PostgreSQL. Status -> `TRANSCRIBED`.
3. **Auto-clip** â€” job `detect-clips` (dienqueue oleh `apps/worker` sendiri begitu `transcribe` sukses, bukan oleh `apps/api`) mengirim transcript ke LLM (GPT) untuk memilih 1-3 momen paling menarik/viral-worthy sebagai kandidat klip (start/end timestamp + virality score 0-100 + `hookText`/`hashtags`, lihat "Hook & Hashtag Generator (Fase 5)" di bawah). Menghasilkan daftar kandidat klip. Status -> `CLIPS_DETECTED`.
4. **Caption + Reframe** â€” untuk tiap kandidat klip, `detect-clips` yang sukses langsung enqueue satu job `render-clip` (self-chain, sama seperti `transcribe` -> `detect-clips`). `render-clip` mendeteksi wajah di sepanjang rentang klip (lihat "Smart Reframe (Fase 2)" di bawah), memotong+crop 9:16 video dengan FFmpeg sesuai timestamp klip dan posisi wajah, generate file ASS dari transcript sesuai `Clip.captionStyle` (lihat "Caption Styling (Fase 3)" di bawah; timestamp digeser relatif ke awal klip), lalu burn-in sebagai subtitle ke video hasil crop. `Clip.outputUrl` di-set setelah render sukses; `Video.status` -> `RENDERED` baru setelah **semua** klip milik video tersebut selesai di-render (bukan begitu klip pertama selesai).
5. **Download** â€” `apps/web` polling `GET /videos/:id` tiap 2 detik sampai status `RENDERED` atau `FAILED`, lalu menampilkan daftar klip dengan link `GET /clips/:id/download` (`apps/api` streaming file dari object storage, key-nya ada di `Clip.outputUrl`).
6. **Retry** â€” kalau status `FAILED`, `apps/web` menampilkan tombol Retry yang manggil `POST /videos/:id/retry`. Tahap yang di-retry disimpulkan dari data yang sudah ada (bukan dari marker "gagal di tahap mana" yang disimpan terpisah): belum ada `TranscriptSegment` -> retry `transcribe`; ada segment tapi belum ada `Clip` -> retry `detect-clips`; ada `Clip` tapi sebagian belum punya `outputUrl` -> retry `render-clip` **hanya** untuk klip yang belum itu (tiap klip render independen, jadi satu klip gagal tidak berarti klip lain ikut di-retry). Aman disimpulkan begitu karena `transcribe`/`detect-clips` masing-masing nulis datanya sekaligus naikkan status di step yang sama â€” kalau job-nya gagal (masuk `catch`), data tahap itu belum sempat ketulis sama sekali. Lihat `VideosService.retry`.

Setiap tahap adalah job terpisah di BullMQ (bukan satu job monolitik) agar retry granular per-tahap dan agar FFmpeg cluster bisa discale independen dari proses ASR.

## Timeline Editor (Fase 1 pasca-MVP)

Route `/videos/:id/edit` di `apps/web` â€” fine-tune hasil auto-clip sebelum download, tanpa perlu re-upload atau menunggu pipeline ulang dari awal.

- **Preview**: `<video>` (stream source asli via `GET /videos/:id/source`, bukan hasil render) + `<canvas>` overlay di atasnya untuk caption, di-redraw tiap `requestAnimationFrame` mengikuti `video.currentTime`. **Approximate, bukan pixel-perfect** terhadap hasil burn-in libass FFmpeg (font/posisi/outline sederhana) â€” tetap approximate setelah Fase 3 (custom caption styling) juga: preview canvas belum ikut menampilkan efek karaoke/bold-highlight per-kata, cuma teks polos bold putih apa pun `captionStyle`-nya. Membangun shared style engine antara canvas dan ASS override FFmpeg dianggap scope creep untuk fase ini â€” lihat "Caption Styling (Fase 3)" di bawah.
- **Timeline**: track klip (bar per klip hasil `detect-clips`, drag handle start/end untuk klip yang sedang dipilih) + track caption (transcript segment yang overlap klip terpilih, visual saja, tidak draggable). Dibangun dengan div ber-posisi absolut + persentase (bukan SVG/canvas) untuk timeline-nya sendiri â€” cuma overlay caption di atas video yang pakai canvas.
- **State**: Zustand (`apps/web/lib/timelineStore.ts`) â€” cukup ringan untuk state timeline (klip + draft trim + flag dirty/saving/rendering per klip) tanpa perlu Redux penuh.
- **Trim manual TIDAK auto re-render**: `PATCH /clips/:id` (`ClipsService.update`) cuma update `startTime`/`endTime` di DB. Render ulang adalah aksi eksplisit terpisah (`POST /clips/:id/render`) supaya geser slider tidak buang compute FFmpeg tiap perubahan kecil.
- **Re-render reuse job `render-clip` yang sudah ada**, bukan job baru â€” `render-clip.worker.ts` sudah generik atas start/end dan overwrite key `renders/<clipId>.mp4` yang sama, jadi tidak ada bedanya secara teknis antara render pertama (dari `detect-clips`) dan re-render manual (dari editor). `ClipsService.render` men-clear `Clip.outputUrl` ke `null` **sebelum** enqueue â€” bukan cuma dibiarkan stale â€” supaya dua hal yang sudah ada otomatis benar tanpa ubah kode lain: tampilan "Rendering..." di dashboard (sudah ada untuk `downloadUrl: null`) langsung kepakai, dan kalau render-clip job ini gagal, `VideosService.retry`'s pengecekan "klip tanpa `outputUrl` perlu di-render lagi" tetap akurat (bukannya mengira klip ini masih fine karena `outputUrl` lama belum dihapus).
- **`GET /videos/:id/source`** streaming source video (bukan hasil render) dengan dukungan HTTP Range (`packages/storage`'s `getObjectStreamRange`) supaya `<video>` bisa scrub/seek tanpa download seluruh file dulu â€” `<video crossOrigin="use-credentials">` di frontend supaya cookie sesi ikut terkirim cross-origin (api beda port dari web). Proxy lewat `apps/api` (bukan presigned URL langsung ke R2), konsisten dengan keputusan storage yang sudah ada: client tidak pernah akses bucket langsung.
- **`GET /videos/:id/transcript`** endpoint terpisah dari `findOne()`/`GET /videos/:id` â€” sengaja tidak digabung supaya endpoint yang di-polling tiap 2 detik (upload progress, dashboard) tidak ikut membawa payload transcript (bisa banyak baris untuk video panjang) padahal cuma editor yang butuh teksnya.
- **`filterSegmentsForClip`** (overlap-filter transcript-untuk-rentang-klip) diekstrak ke `packages/shared/src/utils/transcript.ts` â€” sebelumnya duplikat identik di `detect-clips.worker.ts` dan `VideosService.retry`; ClipsService.render jadi pemakai ketiga, titik di mana duplikasi lebih dari cukup untuk diekstrak jadi util bersama.

## Smart Reframe (Fase 2 pasca-MVP)

Setiap `render-clip` job sekarang men-crop video ke 9:16 mengikuti wajah yang paling menonjol di frame, bukan sekadar center-crop statis dari aspect ratio asli.

- **Inline di job `render-clip` yang sudah ada, bukan job BullMQ baru** (`detect-faces`) â€” deteksi wajah adalah sub-step di awal `render-clip` (posisinya setara dengan `buildSrt()` yang sudah ada: dihitung, dipakai, lalu buang, tidak disimpan lintas job). Tidak perlu schema baru di `Clip`, tidak perlu cabang baru di `VideosService.retry` â€” kalau `render-clip` gagal (termasuk karena deteksi wajah), seluruh job itu yang di-retry sebagai satu unit, sama seperti perilakunya sebelum fase ini.
- **MediaPipe dijalankan lewat subprocess Python asli** (`apps/worker/scripts/detect_faces.py`, dipanggil dari `apps/worker/src/faceDetection.ts` via `child_process` â€” pola yang sama persis dengan `apps/worker/src/ffmpeg.ts` manggil binary `ffmpeg`), bukan lewat `@mediapipe/tasks-vision` (WASM/Node). MediaPipe secara resmi paling matang di Python/browser, bukan Node server-side. Pakai **Tasks API** (`mediapipe.tasks.python.vision.FaceDetector`) â€” API lama `mp.solutions.face_detection` sudah **dihapus** di rilis mediapipe terbaru (0.10.x). Tasks API butuh file model `.tflite` terpisah (tidak dibundel di pip package) â€” di-download dari model zoo MediaPipe, disimpan di `apps/worker/models/` (gitignored, bukan aset yang di-commit; lihat README.md untuk cara download-nya).
- **`node:20-alpine` -> `node:20-slim` (Debian/glibc) untuk image `apps/worker`** â€” MediaPipe's PyPI wheels **tidak ada build musl (Alpine) sama sekali** (`pip install mediapipe` di Alpine langsung gagal "No matching distribution"). `opencv-python-headless` juga butuh beberapa shared library sistem meski headless (`libxcb1 libxext6 libsm6 libgl1 libglib2.0-0 libgomp1 libxrender1 libegl1 libgles2` â€” ditemukan lewat trial-and-error terhadap container asli, bukan disalin dari satu sumber tunggal) dan `pip install --break-system-packages` (Debian 12 menerapkan PEP 668 untuk system Python).
- **Deteksi wajah tidak boleh menggagalkan job** â€” `detectFaces()` di-try/catch dalam `buildReframePlan()` (`render-clip.worker.ts`); kalau subprocess Python gagal karena alasan APAPUN (bukan cuma "tidak ada wajah", tapi juga mis. environment bermasalah), fallback ke center-crop statis, bukan melempar error. Ini perluasan alami dari requirement "jangan gagal cuma karena tidak ada wajah" ke "jangan gagal karena mekanisme deteksinya sendiri bermasalah".
- **Heuristik "wajah paling menonjol", bukan deteksi speaker aktif sungguhan** â€” kalau ada lebih dari satu wajah di satu frame sampel, dipilih yang bounding box-nya paling besar (proxy untuk "yang bicara"). Deteksi speaker aktif sungguhan (perlu Face Landmarker + korelasi audio) di luar scope fase ini.
- **Sampling ~1 frame/detik** (`FACE_SAMPLE_INTERVAL_SECONDS` di `apps/worker/src/faceDetection.ts`) â€” klip auto-detect sudah dibatasi 5-60 detik oleh prompt LLM `detect-clips`, jadi maksimal ~60 titik sampel per klip. Python baca frame langsung lewat OpenCV `VideoCapture.set(CAP_PROP_POS_MSEC, ...)` (seek langsung, bukan ekstrak semua frame ke file gambar dulu) â€” jadi `reserveScratchPath` yang sudah ada **tetap cukup dipakai apa adanya** tanpa perlu direktori scratch baru untuk frame; satu-satunya scratch file baru adalah command file `sendcmd` (lihat di bawah).
- **Interpolasi crop-path dihitung di TypeScript** (`apps/worker/src/reframe.ts`, step lebih rapat â€” `CROP_PATH_STEP_SECONDS` 0.2 detik â€” daripada sampling deteksi wajah), bukan sebagai satu ffmpeg expression yang panjang. Hasilnya di-generate jadi file command `sendcmd` (satu baris per titik interpolasi: `TIME crop@reframe x X, crop@reframe y Y;`) yang dipasang sebelum filter `crop` yang di-tag `@reframe` di filtergraph FFmpeg (`crop@reframe=w=W:h=H:x=X0:y=Y0`) â€” diverifikasi langsung dengan video test pattern sungguhan (posisi crop terbukti benar-benar bergerak, bukan cuma diam di X0/Y0).
- **Fallback ke center-crop statis** (tanpa `sendcmd`, filter `crop=` polos) kalau tidak ada wajah terdeteksi di **satu pun** sampel sepanjang klip (`buildCropPath()` return `null`) â€” bukan cuma error handling, tapi juga menghindari generate crop-path yang sebenarnya konstan (percuma).
- **Crop cuma gerak di satu sumbu** â€” untuk source landscape (kasus paling umum, `sourceAspect > 9/16`), tinggi penuh dipertahankan dan cuma lebar yang di-crop, jadi cuma X yang bergerak (Y tetap 0); untuk source yang sudah portrait, sebaliknya. `computeCropDimensions()` selalu membulatkan ke angka genap (requirement `libx264`/`yuv420p`, menolak lebar/tinggi ganjil).

## Caption Styling (Fase 3 pasca-MVP)

Tiap `Clip` punya preset caption style (`CaptionStyle`: `DEFAULT` | `KARAOKE` | `BOLD_HIGHLIGHT`) yang menentukan cara `render-clip` men-styling subtitle burn-in-nya.

- **SRT diganti ASS/SSA sepenuhnya** (`apps/worker/src/subtitles.ts`'s `buildAss()`, menggantikan `buildSrt()` yang lama di `ffmpeg.ts`) â€” untuk **semua** preset termasuk `DEFAULT`, bukan cuma `KARAOKE`/`BOLD_HIGHLIGHT`, supaya cuma ada satu code path render (tidak ada percabangan format subtitle di `renderClip()`). `renderClip()`'s parameter di-rename `srtPath` -> `subtitlesPath` mengikuti perubahan ini. FFmpeg's `subtitles=` filter yang sama persis dipakai untuk ASS maupun SRT (libass auto-detect format dari isi file), jadi tidak ada perubahan di sisi `ffmpeg.ts` selain nama parameter.
- **`CaptionStyle` disimpan sebagai enum field langsung di `Clip`** (bukan model `CaptionStyle` terpisah) â€” preset tetap (bukan custom style bebas per-user), jadi kolom enum lebih dari cukup. Di-set lewat `PATCH /clips/:id` yang sudah ada dari Fase 1 (extend `UpdateClipDto`), **tidak auto re-render** â€” sama seperti trim manual, user klik render eksplisit lagi lewat `POST /clips/:id/render` (`ClipsService.update`/`.render`).
- **Word-level timestamp dari Whisper cuma berlaku ke depan, tidak di-backfill** â€” `transcribe.worker.ts` sekarang minta `timestamp_granularities: ['word', 'segment']` (satu API call yang sama, tidak nambah cost/panggilan terpisah) dan bucket tiap word ke segment yang memuat range waktunya, disimpan di `TranscriptSegment.words` (kolom `Json?`, nullable). Video yang sudah ditranskrip sebelum Fase 3 tidak punya data ini â€” kalau video begitu di-render dengan preset `KARAOKE`, `buildAss()` fallback ke teks polos per-segment yang word data-nya kosong (bukan gagal); tidak ada UI yang secara eksplisit mem-block pemilihan `KARAOKE` untuk video lama, cukup mengandalkan fallback graceful ini.
- **`KARAOKE` pakai fitur karaoke native ASS (`\k` tag)** â€” per-word `\k<centiseconds>` di dalam satu `Dialogue` line, cocok persis dengan kebutuhan "highlight kata yang sedang diucapkan secara sinkron" karena itu memang fitur bawaan format ASS untuk lirik karaoke. Perlu **dua `Style` ASS terpisah** (`Default` dan `Karaoke`, sama font/posisi, beda `PrimaryColour`/`SecondaryColour`) â€” `\k`'s pewarnaan (Secondary sebelum durasinya habis, Primary sesudahnya) berlaku ke `Style` yang direferensikan baris itu, dan baris **tanpa** tag `\k` selalu tampil di `PrimaryColour`-nya `Style` itu; kalau cuma ada satu `Style` bersama, preset `DEFAULT`/`BOLD_HIGHLIGHT` (tidak pakai `\k`) ikut kebagian warna aksen karaoke, bukan warna putih polos yang dimaksud. Segment `KARAOKE` yang word data-nya kosong pun sengaja dirender pakai `Style: Default` (bukan `Style: Karaoke`) supaya fallback-nya benar-benar netral, bukan solid warna aksen.
- **Heuristik keyword untuk `BOLD_HIGHLIGHT`, bukan input manual atau LLM** â€” kata yang mengandung angka/persentase, ALL-CAPS, atau diapit tanda kutip di-bold + diwarnai lewat override tag `{\b1\c...}...{\r}` inline di teks (`highlightKeywords()`). Ekstraksi keyword yang lebih pintar (semantik, bukan pattern-matching) ditunda ke Fase 5 (hook generator), yang memang sudah dalam roadmap untuk nambah LLM call baru. Beda dengan `KARAOKE`, preset ini **tidak butuh word-level timestamp** â€” cuma butuh teks segment biasa â€” jadi tetap berfungsi penuh untuk video lama.
- **Diverifikasi visual langsung** (bukan cuma unit test) â€” `buildAss()`'s output asli di-burn beneran lewat FFmpeg ke frame video test, dan hasilnya di-screenshot: kata angka/CAPS terbukti benar-benar bold+kuning untuk `BOLD_HIGHLIGHT`, dan highlight `KARAOKE` terbukti bergerak kata-per-kata seiring waktu (frame di t=3.1s/3.6s/4.3s masing-masing menunjukkan 1/2/3 kata yang sudah "ke-highlight").

## Observability (Fase 4 pasca-MVP)

`apps/api` dan `apps/worker` mengirim exception ke Sentry (`@sentry/node`) â€” sebelum fase ini, kegagalan job BullMQ cuma tercatat di log lokal container, tidak ada jejak terpusat yang gampang ditelusuri lintas deploy.

- **`SENTRY_DSN` opsional, bukan wajib** â€” beda dari `DATABASE_URL`/`STORAGE_*`/dst yang fail-fast di boot kalau kosong. `Sentry.init({ dsn: undefined })` cuma men-disable SDK-nya (semua `captureException` jadi no-op) tanpa throw, jadi aman dibiarkan kosong di dev lokal. Ditambahkan ke `apps/api/src/config/env.validation.ts` sebagai `@IsOptional()` (bukan required) dan didokumentasikan (bukan divalidasi, karena memang tidak perlu) di `apps/worker/src/env.ts`.
- **`apps/worker`: `Sentry.captureException(error, { tags: { videoId, clipId? } })`** di catch block masing-masing dari 3 job handler (`transcribe`, `detect-clips`, `render-clip`) â€” persis di titik yang sama dengan `console.error` yang sudah ada. Cuma tag ID yang dikirim, **tidak pernah** isi transcript, source video, atau `OPENAI_API_KEY`/`STORAGE_*` â€” sejalan dengan log `console.error` yang sudah ada sejak awal (cuma template string berisi ID). Fallback yang sudah "expected"/tidak menggagalkan job (mis. deteksi wajah gagal di Fase 2, fallback ke center-crop) **tidak** dilaporkan ke Sentry â€” cuma kegagalan yang benar-benar menggagalkan job yang masuk.
- **`apps/api`: `SentryExceptionFilter` (extends `BaseExceptionFilter`, `@Catch()` global)**, bukan integrasi otomatis `@sentry/nestjs` â€” cukup laporkan exception yang benar-benar tak terduga (bug), bukan full APM/route tracing yang tidak diminta fase ini. Filter ini cuma melapor ke Sentry untuk exception yang BUKAN `HttpException` **atau** `HttpException` dengan status >= 500 â€” `NotFoundException`/`BadRequestException`/dst (alur kontrol normal, bukan bug) sengaja tidak dilaporkan. Setelah lapor, selalu delegasi ke `super.catch()` supaya response HTTP (shape/status code) identik dengan sebelum filter ini ada â€” **hanya** exception object yang dikirim ke Sentry, tidak pernah request/response (yang bisa bawa cookie sesi/JWT).
- **`new SentryExceptionFilter(httpAdapter)` â€” bukan `HttpAdapterHost`-nya langsung** â€” `BaseExceptionFilter` manggil method macam `isHeadersSent()` langsung di objek yang dikasih ke constructor-nya, jadi butuh `app.get(HttpAdapterHost).httpAdapter` (adapter asli), bukan wrapper `HttpAdapterHost`-nya. Ini bug yang sempat kejadian beneran saat implementasi (lolos dari unit test yang nge-mock `BaseExceptionFilter`, ketahuan pas verifikasi E2E asli terhadap request 404/400 sungguhan) â€” pengingat kenapa exception filter yang di-mock unit test-nya tetap butuh dicoba beneran lewat request HTTP asli.
- **Sentry.init() dipanggil setelah titik yang sudah ada untuk load env**, bukan sebelumnya â€” `apps/worker/src/main.ts` sudah panggil `dotenv.config()` manual di baris paling atas (pola lama), jadi `initSentry()` taruh persis setelah itu. `apps/api` TIDAK pakai dotenv manual (`ConfigModule.forRoot()` yang urus env), jadi `initSentry()` dipanggil di `bootstrap()` setelah `NestFactory.create(AppModule)` selesai (titik pertama `process.env.SENTRY_DSN` dijamin sudah ke-populate) â€” bukan menambah `dotenv.config()` manual baru ke `apps/api` cuma demi Sentry, supaya tidak ada dua mekanisme load env yang tumpang tindih di app yang sama.

## Hook & Hashtag Generator (Fase 5 pasca-MVP)

`detect-clips`'s LLM call yang sudah ada (scoring virality) sekaligus menghasilkan `hookText` (saran baris pembuka ~3 detik pertama) dan `hashtags` (array tanpa `#`) per kandidat klip â€” bukan job/LLM call baru terpisah.

- **Extend prompt & JSON schema yang sudah ada, bukan job baru** â€” `RESPONSE_FORMAT`'s per-candidate schema nambah `hookText`/`hashtags` sebagai field required di samping `startTime`/`endTime`/`viralityScore` yang sudah ada; satu system prompt dan satu API call yang sama menghasilkan keduanya sekaligus. Dipilih supaya tidak nambah biaya+latency LLM call kedua untuk sesuatu yang bisa digabung sekali jalan (LLM sudah baca full transcript untuk scoring, konteks yang sama persis dibutuhkan untuk menulis hook/hashtag yang relevan).
- **Bahasa hookText/hashtags mengikuti bahasa transkrip video**, bukan dipaksa Inggris atau Indonesia â€” instruksi eksplisit di system prompt ("Write both hookText and hashtags in the same language as the transcript"), diverifikasi langsung end-to-end (transkrip video test berbahasa Inggris menghasilkan hook/hashtag berbahasa Inggris).
- **`hashtags` disanitasi di kedua sisi yang bisa menulisnya** â€” `sanitizeHashtags()` (`packages/shared/src/utils/hashtags.ts`, dipakai `detect-clips.worker.ts` dan `ClipsService.update`) strip leading `#` dan buang entry kosong/whitespace. LLM sudah diinstruksikan "no leading #", tapi tetap disanitasi (belt-and-suspenders, jangan percaya model sepenuhnya ikut instruksi format); demikian juga input manual lewat `PATCH /clips/:id` disanitasi dengan fungsi yang sama supaya data konsisten terlepas dari jalur mana yang menulisnya â€” bug ini sempat kejadian beneran (PATCH manual awalnya tidak disanitasi, ketahuan pas tes E2E asli mengirim `"#stripme"` dan hasilnya kesimpan dengan `#`-nya).
- **`Clip.hookText`/`Clip.hashtags` disimpan langsung di model `Clip`** (bukan tabel terpisah) â€” `hookText String?` (nullable; dalam praktiknya selalu terisi dari `detect-clips` karena field required di schema LLM, nullable cuma jaga-jaga) dan `hashtags String[] @default([])` (array native Postgres lewat Prisma, default array kosong bukan nullable supaya caller tidak perlu null-check). Migration baru (`add_clip_hook_and_hashtags`), sama pola dengan field `Clip` lain yang sudah ada.
- **Purely metadata, tidak memengaruhi hasil render** â€” beda dari `captionStyle` (Fase 3) yang benar-benar mengubah video hasil FFmpeg, `hookText`/`hashtags` tidak pernah dibaca oleh `render-clip`/`ffmpeg.ts`. Edit lewat `PATCH /clips/:id` (extend `UpdateClipDto` yang sudah ada dari Fase 1/3) langsung tersimpan tanpa perlu trigger render ulang sama sekali â€” beda dari trim/caption style yang tetap butuh klik "Render" eksplisit karena mengubah file video.
- **Ditampilkan di dua tempat**: read-only di dashboard (`apps/web/app/dashboard/page.tsx`'s clip list â€” "Clip Gallery") dan editable di Timeline Editor (`TimelineEditor.tsx`, pakai `setHookText`/`setHashtags` di `timelineStore.ts`, disimpan lewat tombol "Save" yang sama dengan trim/caption style). Input hashtags di editor pakai `<input>` **uncontrolled** (`key={clipId}` + `defaultValue` + commit di `onBlur`) bukan controlled biasa â€” controlled value yang di-derive ulang dari `hashtags.join(' ')` di tiap keystroke akan langsung membuang spasi/koma trailing yang baru diketik user, bikin tidak mungkin mulai kata kedua (ketahuan langsung waktu ditulis, sebelum sempat jadi bug nyata).

## Publish Center â€” Social Account Connect (Fase 6a pasca-MVP)

Fase 6 (publish scheduler multi-platform + analytics) dipecah jadi sub-fase; 6a cuma mengurus koneksi akun sosmed (connect/refresh/disconnect token) â€” **belum ada aksi publish sama sekali**. Ini sengaja diisolasi dulu karena OAuth per-platform adalah bagian paling berisiko/rumit dari seluruh Fase 6, dan lebih murah dibuktikan benar sendirian sebelum dibangun apa pun di atasnya.

- **YouTube Shorts dipilih sebagai platform pertama** (dari TikTok/Instagram Reels/YouTube Shorts) â€” OAuth2 Google standar, API paling matang, tidak ada gate app-review untuk scope upload dasar (beda dari TikTok's Content Posting API yang butuh audit dulu untuk direct-public-post, atau Instagram Reels yang butuh akun IG Business/Creator + flow publish 2 langkah via Meta Graph API). Prioritas platform berikutnya (Fase 6d) adalah keputusan bisnis, bukan teknis â€” lihat CLAUDE.md's breakdown history di percakapan, bukan didokumentasikan di sini.
- **Model baru `SocialAccount`** (`userId`, `platform` enum, `platformAccountId`, `displayName`, `accessToken`/`refreshToken` terenkripsi, `tokenExpiresAt`) â€” satu user bisa connect banyak akun, termasuk beberapa akun di platform yang sama (mis. 2 channel YouTube berbeda); unique constraint di `(userId, platform, platformAccountId)` supaya reconnect akun yang sama meng-upsert, bukan bikin baris duplikat.
- **Token dienkripsi app-level (AES-256-GCM), bukan mengandalkan enkripsi native provider hosting** â€” `apps/api/src/social/token-encryption.util.ts`, key dari env var baru `TOKEN_ENCRYPTION_KEY` (`openssl rand -hex 32`, sama pola generate dengan `JWT_SECRET`). Disimpan sebagai `"<iv>:<authTag>:<ciphertext>"` hex. Dipilih supaya portable ke provider hosting mana pun, tidak terikat fitur encryption-at-rest Postgres/cloud tertentu.
- **`GOOGLE_OAUTH_CLIENT_ID`/`GOOGLE_OAUTH_CLIENT_SECRET`/`TOKEN_ENCRYPTION_KEY` semua opsional di boot** (pola sama dengan `SENTRY_DSN` di Fase 4) â€” sisa `apps/api` harus tetap jalan normal buat semua orang yang belum setup Google Cloud OAuth client-nya sendiri. Konfigurasi yang kurang baru jadi error (503 "YouTube integration is not configured") persis di titik `GET /social/youtube/connect` benar-benar dipanggil, bukan bikin seluruh app gagal boot. `TOKEN_ENCRYPTION_KEY` yang kosong juga tidak diam-diam fallback ke key hardcoded manapun â€” `token-encryption.util.ts` sengaja throw error jelas di titik enkripsi/dekripsi pertama kali dipakai.
- **OAuth `state` param ditandatangani pakai `JwtModule` instance terpisah dari `AuthModule`** (`SocialModule` register `JwtModule.registerAsync()` sendiri, `JWT_SECRET` yang sama tapi expiry beda jauh lebih pendek â€” 10 menit) â€” dipakai murni buat mengaitkan callback dari Google balik ke user mana yang connect, bukan token sesi. `GET /social/youtube/callback` sengaja **tidak** dipasangi `JwtAuthGuard` â€” sumber kebenaran soal "siapa yang connect" adalah `state` yang di-decode, bukan cookie sesi (yang bisa saja sudah expired/logout di antara redirect ke Google dan baliknya).
- **`GET /social/youtube/connect` bukan `fetch()` dari `apps/web`, tapi `<a href>` (top-level browser navigation)** â€” OAuth butuh redirect browser sungguhan ke Google, yang tidak bisa dilakukan `fetch()`. Cookie sesi tetap terkirim di navigasi top-level ini (`SameSite=Lax` mengizinkan itu, dan `apps/web`/`apps/api` di dev sama-sama di bawah "site" `localhost`), jadi `JwtAuthGuard` tetap bisa resolve user di endpoint `connect` (beda dengan `callback`, lihat poin di atas).
- **`OAuth2Client` dari `google-auth-library`** (official Google client) dipakai untuk mekanisme OAuth (authorize URL, token exchange, refresh, revoke) â€” bukan hand-roll raw HTTP call ke endpoint OAuth Google, karena kesalahan kecil di kode OAuth (mis. parameter yang salah/kurang) adalah kelas bug yang mahal (auth diam-diam gagal atau lebih parah, celah keamanan). Fetch channel info (buat display name di UI) pakai `fetch()` polos ke YouTube Data API - satu endpoint sederhana, tidak perlu SDK `googleapis` yang jauh lebih berat cuma untuk itu.
- **Diverifikasi nyata sejauh yang bisa** â€” `apps/api` boot normal tanpa `GOOGLE_OAUTH_CLIENT_ID`/`SECRET`/`TOKEN_ENCRYPTION_KEY` sama sekali (state dev paling realistis sebelum user setup Google Cloud sendiri), `GET /social/accounts` mengembalikan `[]` yang benar, `GET /social/youtube/connect` mengembalikan 503 yang jelas. Seluruh lifecycle connect->reconnect(upsert)->list->deteksi-expiry->disconnect diverifikasi langsung terhadap Postgres asli dengan enkripsi AES-256-GCM asli (bukan mock). **Batasan yang diketahui**: round-trip OAuth sungguhan ke server Google (authorize -> consent -> exchange -> refresh -> revoke) tidak bisa diverifikasi end-to-end karena butuh kredensial OAuth client asli dari Google Cloud Console yang cuma bisa dibuat manual oleh user (sama seperti `SENTRY_DSN` di Fase 4) â€” mekanisme HTTP-nya sendiri (lewat `google-auth-library`) sudah diuji unit test dengan mock terhadap client library resminya.
- **Belum ada job BullMQ baru** â€” 6a murni operasi HTTP sinkron di `apps/api` (OAuth handshake), tidak menyentuh `apps/worker` sama sekali. `getValidAccessToken()` (proses refresh) ada sebagai method service yang teruji tapi belum dipanggil dari endpoint HTTP manapun â€” fase publish-clip berikutnya (6b) yang akan benar-benar memakainya.

## Publish Center â€” Manual "Publish Now" (Fase 6b pasca-MVP)

Tombol "Publish to YouTube" di dashboard (Clip Gallery) untuk klip yang sudah selesai di-render â€” meng-upload klip itu sungguhan ke channel YouTube user yang sudah connect (Fase 6a), lewat job BullMQ baru `publish-clip`. Publish selalu langsung begitu tombol diklik di fase ini â€” lihat "Publish Center â€” Scheduling (Fase 6c pasca-MVP)" di bawah untuk publish di waktu yang dijadwalkan.

- **`packages/social` (package baru)** â€” OAuth client (`YouTubeOAuthClient`), enkripsi token (`encryptToken`/`decryptToken`), `resolveAccessToken()` (cek expiry + refresh kalau perlu), dan `uploadYouTubeVideo()` dipindah/ditaruh di sini, bukan di `apps/api/src/social/` seperti Fase 6a. Alasannya: `publish-clip` job di `apps/worker` butuh access token YouTube yang valid (decrypt + refresh), tapi `apps/worker` tidak bisa import langsung dari `apps/api` (dua deployable app terpisah, cuma `packages/*` yang shared antar keduanya). Ini pengecualian sadar terhadap prinsip "ekstrak ke shared code di duplikasi ke-3, bukan ke-2" yang biasa dipakai proyek ini â€” OAuth/enkripsi token adalah kelas bug berisiko tinggi (drift halus di sini = celah keamanan atau job publish yang diam-diam rusak), jadi lebih murah dibagi sejak awal daripada didupilkasi sekali dulu. `apps/api`'s `SocialAccountsService`/`SocialController` sekarang import dari `@viral-clip-app/social`, bukan file lokal lagi.
- **`googleapis` dipakai untuk `videos.insert` yang sesungguhnya** (bukan hand-roll protokol resumable upload) â€” konsisten dengan keputusan Fase 6a pakai `google-auth-library` resmi untuk OAuth ("implementasi custom yang subtly-wrong untuk protokol multi-request itu kelas bug yang jauh lebih mahal daripada bobot dependency-nya"). **`googleapis` membundel salinan internal `google-auth-library` sendiri** yang tipe `OAuth2Client`-nya secara struktural tidak kompatibel dengan package `google-auth-library` standalone yang dipakai `youtube-oauth.client.ts` untuk alur connect/refresh/revoke â€” walau keduanya resolve ke versi yang sama-sama valid secara semver. Solusinya bukan memaksa satu helper `authorizedClient()` dipakai bersama, tapi memisahkan: `youtube-upload.client.ts` (upload) membangun objek auth-nya sendiri lewat `google.auth.OAuth2` dari `googleapis`'s namespace sendiri, sementara `youtube-oauth.client.ts` (connect/refresh/revoke) tetap pakai `OAuth2Client` standalone â€” dua konstruksi terpisah yang masing-masing konsisten secara tipe dengan library yang dia pakai.
- **Model baru `PublishRecord`** (`clipId`, `socialAccountId`, `status` enum `QUEUED`/`PUBLISHING`/`PUBLISHED`/`FAILED`, `platformPostId`, `errorMessage`, `publishedAt`, plus `scheduledAt DateTime?` yang sengaja ditambahkan sekarang walau selalu `null` di 6b â€” supaya Fase 6c (scheduling) tidak butuh migration kedua cuma untuk kolom itu) â€” satu baris per **percobaan** publish klip ke satu akun sosmed (retry BullMQ mengubah status baris yang sama, bukan bikin baris baru). Relasi ke `Clip` dan `SocialAccount`, keduanya `onDelete: Cascade`.
- **`ClipsService.publish()` bikin baris `PublishRecord` secara sinkron (status `QUEUED`) sebelum enqueue job** â€” sama seperti `render()`'s pola "clear `outputUrl` dulu sebelum enqueue" dari Fase 1: supaya UI langsung bisa nampilin status "Queued to publish..." begitu tombol diklik, tanpa nunggu job benar-benar mulai diproses worker.
- **Job `publish-clip` cuma bawa `{ publishRecordId }` di payload-nya**, bukan snapshot clipId/socialAccountId/dll â€” konsisten dengan pola job lain di proyek ini (payload minimal, re-fetch state terkini dari Postgres di worker) supaya tidak ada risiko data enqueue-time yang sudah basi dipakai saat job benar-benar jalan. Worker fetch `PublishRecord` beserta relasi `clip`+`socialAccount`-nya sekaligus lewat satu `include`.
- **`publish-clip` adalah job BullMQ pertama di proyek ini yang pakai retry otomatis bawaan BullMQ** (`attempts: 3`, `backoff: { type: 'exponential', delay: 30_000 }`, dikonfigurasi di `ClipsService.publish()` sebagai argumen ketiga `queue.add()`) â€” beda dari `transcribe`/`detect-clips`/`render-clip` yang semuanya gagal sekali lalu nunggu user klik Retry manual. Dibedakan sengaja karena panggilan ke API platform eksternal (YouTube) punya kelas kegagalan transient yang tidak dimiliki job-job lain (rate limit, 5xx sementara) â€” user tidak perlu tahu/ikut campur soal itu. `PublishRecord.status` cuma di-set ke `FAILED` (dengan `errorMessage`) di attempt **terakhir** (`job.attemptsMade + 1 >= job.opts.attempts`) â€” kalau masih ada retry tersisa, baris dibiarkan `PUBLISHING` supaya UI tidak salah nampilin "gagal" padahal masih ada percobaan otomatis berikutnya yang jalan.
- **Default `privacyStatus: 'unlisted'`** untuk video yang di-upload â€” ini video sungguhan yang benar-benar ke-publish ke channel YouTube user begitu tombol "Publish now" diklik. `public` langsung dianggap terlalu berisiko untuk fase yang tujuannya "membuktikan jalur upload-ke-platform bekerja" (klip yang salah pilih atau caption yang belum pas bisa langsung live tanpa jaring pengaman), sementara `private` dianggap mengalahkan tujuan "publish now" itu sendiri (user tetap harus ubah manual visibility-nya lewat YouTube Studio). Tidak ada opsi visibility yang bisa diubah user di fase ini â€” keputusan produk untuk fase berikutnya kalau dibutuhkan.
- **Title dari `Clip.hookText`** (fallback ke `"Clip <id>"` kalau kosong) dan **description dari `Clip.hashtags`** (di-join jadi `#tag1 #tag2 ...`) â€” field yang sama yang sudah ada dari Fase 5, tidak ada input title/description terpisah untuk publish di fase ini.
- **`GET /videos/:id` dan `PATCH/POST /clips/:id/*` sekarang ikut mengembalikan `Clip.publishRecords[]`** (`toSharedPublishRecord()` di `apps/api/src/social/publish-record.util.ts`, dipakai `ClipsService` dan `VideosService`) â€” `platform` di-denormalisasi dari `SocialAccount` yang di-join supaya dashboard bisa langsung nampilin "Published to YouTube" tanpa query terpisah. Dashboard polling `GET /videos` yang sudah ada (interval 2 detik, sama seperti transcribe/detect-clips/render-clip) otomatis kepakai buat mantau progres publish juga â€” tidak ada endpoint/mekanisme polling baru.
- **Diverifikasi nyata sejauh yang bisa** â€” lifecycle penuh `PublishRecord` (create -> QUEUED -> job jalan -> PUBLISHING -> PUBLISHED/FAILED) diuji lewat mock jaringan (`uploadYouTubeVideo`/`resolveAccessToken` di-mock) terhadap Postgres asli, termasuk retry/backoff yang benar-benar terpasang di job BullMQ yang di-enqueue dan tagging Sentry (`publishRecordId`/`clipId`/`socialAccountId` saja, tidak pernah token atau isi video). **Batasan yang diketahui**: panggilan `videos.insert` sungguhan ke server YouTube (upload video asli, konfirmasi visibility `unlisted` benar-benar diterapkan di sisi Google) tidak bisa diverifikasi end-to-end tanpa kredensial OAuth + akun YouTube asli milik user â€” sama seperti batasan OAuth round-trip di Fase 6a.

## Publish Center â€” Scheduling (Fase 6c pasca-MVP)

`POST /clips/:id/publish` menerima `scheduledAt` (ISO 8601) opsional â€” kalau diisi (dan waktunya di masa depan), klip dijadwalkan publish nanti alih-alih langsung, lewat `PublishRecord` status baru `SCHEDULED` dan job BullMQ repeatable baru `schedule-publish-clip` di `apps/worker` yang men-poll Postgres.

- **Mekanisme: BullMQ repeatable job yang polling Postgres, bukan BullMQ delayed job** â€” dipilih secara eksplisit di antara dua kandidat karena konsisten dengan prinsip yang sudah ada di proyek ini ("PostgreSQL sebagai source of truth, Redis cuma antrian/cache, bukan penyimpanan permanen" â€” lihat "Keputusan Arsitektur" di bawah). Kalau "jadwal" disimpan sebagai delayed job di dalam Redis (opsi lain yang dipertimbangkan), kehilangan data Redis (restart tanpa persistence, flush, dst) bisa diam-diam menghapus klip yang harusnya publish di masa depan tanpa jejak apa pun. Dengan polling, `PublishRecord.scheduledAt` di Postgres tetap jadi satu-satunya sumber kebenaran â€” Redis kehilangan data cuma menunda job poller berikutnya, tidak pernah menghapus rencana publish. Trade-off yang diterima: presisi waktu publish maksimal mundur sejauh interval polling (`POLL_INTERVAL_MS` = 60 detik, `apps/worker/src/workers/schedule-publish-clip.worker.ts`).
- **`PublishStatus` nambah `SCHEDULED`** (state paling awal, sebelum `QUEUED`) â€” dipakai murni untuk klip yang scheduledAt-nya belum tiba. `ClipsService.publish()` sekarang bercabang: `scheduledAt` kosong -> tetap seperti Fase 6b (`status: QUEUED`, langsung `publishClipQueue.add()`); `scheduledAt` terisi (divalidasi harus di masa depan, kalau tidak `BadRequestException`) -> `status: SCHEDULED`, **tidak** enqueue apa pun ke `publish-clip` sampai poller yang men-trigger-nya nanti.
- **Poller meng-klaim baris secara atomik** â€” `schedule-publish-clip.worker.ts`'s processor query semua `PublishRecord` dengan `status: SCHEDULED AND scheduledAt <= now()`, lalu untuk tiap baris jalankan `updateMany({ where: { id, status: SCHEDULED }, data: { status: QUEUED } })` dan cuma `publishClipQueue.add()` kalau `count === 1`. Guard `status: SCHEDULED` di `WHERE` klausa `updateMany` inilah yang membuat klaim atomik â€” walau dua firing job repeatable ini entah bagaimana tumpang tindih (mis. firing sebelumnya masih jalan lambat pas firing berikutnya mulai), baris yang sama tidak akan pernah ke-enqueue dua kali, karena `updateMany` kedua yang mencoba baris yang sudah bukan `SCHEDULED` lagi akan `count: 0`.
- **`PUBLISH_RETRY_OPTIONS` dipindah ke `packages/shared`** (dari const privat di `ClipsService`, Fase 6b) â€” begitu `apps/worker`'s poller juga perlu men-enqueue `publish-clip` dengan retry/backoff yang **persis sama** dengan `ClipsService.publish()`'s immediate-publish path, duplikasi angka retry di dua tempat (rawan drift diam-diam) jadi lebih mahal daripada satu konstanta shared â€” pola yang sama dengan `filterSegmentsForClip`/`sanitizeHashtags` sebelumnya.
- **`apps/worker` sekarang juga punya `Queue` producer untuk `PUBLISH_CLIP`** (`apps/worker/src/queues.ts`'s `publishClipQueue`) â€” di Fase 6b, `publish-clip.worker.ts` adalah leaf job (tidak self-chain ke job lain), jadi `apps/worker` sebelumnya tidak pernah perlu jadi *producer* ke queue itu, cuma consumer. Poller adalah producer pertama di sisi `apps/worker` untuk `PUBLISH_CLIP`.
- **Cancel & reschedule** (`DELETE`/`PATCH /clips/:id/publish/:recordId`) **cuma berlaku selagi `SCHEDULED`** â€” begitu poller sempat meng-klaimnya (jadi `QUEUED` atau lebih lanjut), request ini dianggap "tidak ditemukan" (404), bukan mencoba membatalkan/menghentikan job yang sudah di tangan worker. Scoping `id + clipId + status: SCHEDULED` dicek atomik dalam satu `deleteMany`/`updateMany` (bukan baca-lalu-tulis terpisah) supaya tidak ada race antara pengecekan status dan aksinya sendiri.
- **Cancel meng-hapus baris `PublishRecord`-nya langsung** (bukan status `CANCELED` baru) â€” baris `SCHEDULED` yang dibatalkan belum pernah benar-benar "terjadi" (belum di-enqueue sama sekali), jadi tidak ada nilai historis untuk disimpan; menambah satu enum value lagi cuma untuk kasus ini dianggap scope creep dibanding sekadar menghapus baris rencana yang belum jalan.
- **UI dashboard**: input `datetime-local` + tombol "Schedule" di samping "Publish now" yang sudah ada. Baris `PublishRecord` berstatus `SCHEDULED` ditampilkan beda dari status lain â€” "Scheduled for \<tanggal\>" plus tombol Cancel dan Reschedule (input `datetime-local` inline) â€” sementara status lain (`QUEUED`/`PUBLISHING`/`PUBLISHED`/`FAILED`) tetap pakai tampilan label singkat yang sudah ada dari Fase 6b. Dashboard's polling `GET /videos` yang sudah ada (2 detik) otomatis menangkap transisi `SCHEDULED` -> `QUEUED` begitu poller men-klaimnya, tanpa mekanisme polling baru.
- **Diverifikasi lewat unit test terhadap logic klaim atomik dan validasi tanggal** (mock Prisma/BullMQ) â€” belum ada verifikasi E2E baru terhadap Postgres+Redis asli khusus untuk scheduling di luar yang sudah dicakup Fase 6b's publish-clip pipeline (poller ini cuma menambah satu langkah `SCHEDULED -> QUEUED` sebelum job yang sama persis berjalan).

## Publish Center â€” Platform Kedua: TikTok (Fase 6d pasca-MVP)

Tombol "Connect TikTok" di halaman `/accounts` dan opsi platform TikTok di tombol "Publish to \<platform\>" dashboard â€” `SocialAccount.platform` nambah `TIKTOK`, dan `publish-clip.worker.ts` bercabang upload ke YouTube atau TikTok tergantung platform akun yang dipilih.

- **Mode "Upload to Inbox" (draft), bukan Direct Post â€” keputusan bisnis eksplisit, bukan default teknis.** TikTok Content Posting API punya dua target publish: Direct Post (publish sungguhan ke profil user, tapi butuh app di-audit dulu oleh TikTok â€” approval tidak dijamin dan bisa makan waktu) dan Upload to Inbox (kirim video sebagai draft ke inbox TikTok user, jalan langsung tanpa audit, tapi user tetap harus buka app TikTok dan post manual untuk benar-benar live). Proyek ini memilih Upload to Inbox supaya fitur publish TikTok bisa langsung dipakai tanpa bergantung pada approval TikTok yang tidak pasti â€” konsekuensinya, **"Publish to TikTok" TIDAK benar-benar mem-publish** seperti "Publish to YouTube"; UI secara eksplisit menampilkan "Sent to TikTok â€” open the TikTok app to finish posting" (bukan "Published") untuk `PublishRecord` TikTok yang statusnya `PUBLISHED`, supaya user tidak salah kira klipnya sudah live.
- **Tidak ada SDK Node resmi untuk TikTok** (beda dari Google yang punya `google-auth-library`/`googleapis`) â€” `tiktok-oauth.client.ts` dan `tiktok-upload.client.ts` (keduanya di `packages/social`) hand-roll seluruh HTTP call lewat `fetch()` polos. Ini BUKAN pelanggaran terhadap prinsip "pakai official client library" dari Fase 6a/6b â€” prinsip itu berlaku ketika ada library resmi untuk dihindari hand-roll-nya; untuk TikTok, tidak ada alternatif sama sekali.
- **`resolveAccessToken()` digeneralisasi** â€” sebelumnya (Fase 6b) parameternya secara eksplisit bertipe `YouTubeOAuthClient`. Sekarang menerima `OAuthRefreshClient`, interface minimal (`refreshAccessToken(refreshToken): Promise<{accessToken, refreshToken, expiresAt}>`) yang diimplementasikan baik `YouTubeOAuthClient` maupun `TikTokOAuthClient` â€” supaya logic "cek expiry, refresh kalau perlu" yang sama tetap dipakai bersama lintas platform, bukan didupilkasi atau di-`as any`-kan.
- **`SocialAccountsService` dan `publish-clip.worker.ts` masing-masing punya `clientFor(platform)`/`oauthClientFor(platform)`** â€” satu `switch` kecil yang memetakan `SocialAccount.platform` ke instance client OAuth yang tepat (`YouTubeOAuthClient` atau `TikTokOAuthClient`), dipakai oleh `disconnect()` (revoke), `getValidAccessToken()`/`resolveAccessToken()` (refresh), dan upload. Dua tempat, bukan diekstrak ke `packages/social` â€” sesuai konvensi "ekstrak di duplikasi ke-3" proyek ini, switch 3 baris ini terlalu kecil untuk dijadikan abstraksi bersama.
- **Scope OAuth cuma `user.info.basic` + `video.upload`** (bukan `video.publish`, yang butuh audit "direct public post") â€” `video.upload` adalah scope yang benar untuk Upload to Inbox dan tersedia untuk app yang belum diaudit (walau masih ada batasan sandbox: akun TikTok penguji biasanya perlu didaftarkan sebagai "target user" di TikTok Developer Portal selama app belum lolos App Review dasar).
- **Upload TikTok single-chunk, buffer penuh di memori** (`total_chunk_count: 1`, bukan protokol multi-chunk PUT TikTok yang lengkap) â€” klip di proyek ini dibatasi ~60 detik oleh prompt LLM `detect-clips` (Fase MVP), jadi ukurannya selalu jauh di bawah batas satu PUT. Endpoint init TikTok butuh `video_size` diketahui persis di awal (beda dari `uploadYouTubeVideo()` yang bisa pipe stream langsung ke `googleapis`), jadi stream harus di-buffer penuh dulu sebelum panggilan init.
- **Tidak ada title/caption yang bisa diisi via API untuk mode Inbox** â€” beda dari YouTube (`hookText`/`hashtags` dipetakan ke title/description), endpoint `inbox/video/init` TikTok cuma menerima `source_info`, tidak ada field caption sama sekali di mode ini (cuma tersedia di Direct Post). User yang mengisi caption sendiri saat menyelesaikan posting di app TikTok â€” `hookText`/`hashtags` klip tidak dipakai sama sekali untuk publish TikTok.
- **Diverifikasi lewat unit test terhadap OAuth client, upload client, dan dispatch platform** (mock `fetch`/Prisma/BullMQ) â€” sama seperti batasan Fase 6a/6b, round-trip OAuth sungguhan ke server TikTok dan upload video asli ke inbox TikTok tidak bisa diverifikasi end-to-end tanpa TikTok Developer app + akun TikTok asli milik user.

## Publish Center â€” Platform Ketiga: Instagram Reels (Fase 6d follow-up)

Tombol "Connect Instagram" di `/accounts` dan opsi platform Instagram di tombol "Publish to \<platform\>" dashboard â€” `SocialAccount.platform` nambah `INSTAGRAM`. Beda dari YouTube/TikTok, Instagram punya dua kendala arsitektur baru yang keduanya keputusan eksplisit (bukan default teknis semata): jalur OAuth-nya, dan cara klip diserahkan ke server Meta.

- **Jalur OAuth: Facebook Login + akun IG ditautkan ke Facebook Page (bukan Instagram Login langsung yang lebih baru dari Meta)** â€” dipilih karena lebih matang/terdokumentasi, walau konsekuensinya user WAJIB punya akun Instagram Business/Creator yang ditautkan ke sebuah Facebook Page yang dia kelola (bukan akun IG personal biasa). Sama seperti TikTok, scope `instagram_content_publish` produksi penuh butuh App Review dari Meta; sebelum lolos review, cuma akun yang didaftarkan sebagai Instagram Tester di App Meta yang bisa dipakai.
- **Video diserahkan ke Meta lewat presigned URL langsung dari R2/S3, bukan byte-upload seperti YouTube/TikTok** â€” Instagram Content Publishing API TIDAK punya opsi terima upload byte sama sekali; API-nya cuma bisa "ambil video dari URL publik yang bisa diakses tanpa login" (`video_url` di body `POST /{ig-user-id}/media`). Ini pertama kalinya proyek ini mengekspos sesuatu yang menyerupai link langsung ke bucket â€” sebelumnya SEMUA akses klip lewat endpoint `apps/api` yang butuh sesi login (lihat "Keputusan Arsitektur" â€” storage tidak pernah diekspos publik). `packages/storage`'s `getPresignedDownloadUrl(key, expiresInSeconds)` (pakai `@aws-sdk/s3-request-presigner`) generate URL sementara (15 menit, `INSTAGRAM_PRESIGNED_URL_TTL_SECONDS` di `publish-clip.worker.ts`) yang acak/tidak bisa ditebak dan otomatis kedaluwarsa â€” dipertimbangkan lebih aman/lebih sederhana daripada alternatif "endpoint publik baru di `apps/api` dengan token sekali-pakai" yang juga sempat dipertimbangkan.
- **Model refresh token Facebook beda dari YouTube/TikTok, tapi tetap muat di interface `OAuthRefreshClient` yang sama** â€” Meta tidak punya refresh_token terpisah untuk alur ini; yang ada cuma access token USER long-lived (~60 hari) yang di-re-exchange lewat grant `fb_exchange_token` buat diperpanjang. Tapi token yang benar-benar dipakai untuk panggilan Content Publishing API adalah **Page access token** (didapat dari `/me/accounts`, bukan token user itu sendiri) â€” jadi `SocialAccount.accessToken` menyimpan Page token, dan `SocialAccount.refreshToken` menyimpan token USER long-lived (dipakai `refreshAccessToken()` buat re-derive Page token baru nanti). `InstagramOAuthClient.refreshAccessToken(refreshToken)` tetap mengembalikan bentuk `{accessToken, refreshToken, expiresAt}` yang sama seperti YouTube/TikTok, jadi `resolveAccessToken()` tidak perlu tahu bedanya sama sekali.
- **`fetchAccountInfo()` otomatis pilih Facebook Page PERTAMA yang punya akun Instagram Business tertaut** â€” user dengan banyak Page/akun IG belum bisa memilih lewat UI picker di fase ini; simplifikasi yang sama semangatnya dengan keputusan "klaim job repeatable pertama" di Fase 6c, didokumentasikan sadar sebagai batasan, bukan bug.
- **Container create -> poll status -> publish, dua panggilan API terpisah dengan proses async di antaranya** â€” beda dari YouTube (`videos.insert` satu panggilan sinkron) dan TikTok (init + PUT satu rangkaian cepat), Instagram butuh `POST /{ig-user-id}/media` (bikin container), lalu **polling** `GET /{container-id}?fields=status_code` sampai `FINISHED` (maksimal 5 menit, `POLL_TIMEOUT_MS` di `instagram-upload.client.ts`) sebelum `POST /{ig-user-id}/media_publish` benar-benar bisa jalan. `ERROR`/`EXPIRED` dianggap gagal permanen (tidak di-retry di dalam polling itu sendiri â€” BullMQ's retry/backoff yang sudah ada di level job yang menangani percobaan ulang).
- **Caption Instagram Reels beda dari YouTube's title+description terpisah** â€” cuma ada satu field `caption`, jadi `hookText` dan `hashtags` digabung (`buildCaption()`, pisah baris kosong) alih-alih dipetakan ke dua field terpisah seperti YouTube. Beda juga dari TikTok Upload-to-Inbox yang sama sekali tidak punya field caption â€” Instagram Reels justru API publish sungguhan (`media_publish` yang berhasil = benar-benar live), jadi `hookText`/`hashtags` dipakai penuh, bukan diabaikan.
- **"Published" untuk Instagram artinya publish sungguhan** (beda dari TikTok's "Upload to Inbox") â€” tidak ada override teks khusus di dashboard seperti TikTok, `PUBLISHED` + `platformPostId` (media id Instagram) ditampilkan apa adanya. Tidak ada link ke Reel-nya langsung (media id numerik dari `media_publish` bukan permalink/shortcode publik â€” mengambil permalink butuh satu panggilan Graph API tambahan, ditunda ke fase berikutnya kalau dibutuhkan).
- **Diverifikasi lewat unit test terhadap OAuth client (termasuk model refresh 2-langkah), upload client (container create + polling + publish, pakai fake timer buat polling interval), dispatch platform, dan `getPresignedDownloadUrl`** (mock `fetch`/`@aws-sdk/s3-request-presigner`/Prisma/BullMQ) â€” sama seperti batasan YouTube/TikTok, round-trip OAuth sungguhan ke server Meta dan publish Reel asli tidak bisa diverifikasi end-to-end tanpa Meta app + akun Instagram Business/Facebook Page asli milik user.

## Keputusan Arsitektur

- **BullMQ dipakai untuk semua kerja berat/async** (transcribe, detect-clips, render-clip). API layer tidak pernah menjalankan Whisper atau FFmpeg secara sinkron di request-response cycle.
- **Worker dipisah dari API** supaya FFmpeg cluster dan proses ASR yang CPU/GPU-intensive bisa di-scale terpisah dari layer API yang menangani traffic HTTP.
- **PostgreSQL sebagai source of truth** untuk status job dan metadata video/klip; Redis hanya untuk antrian (BullMQ) dan cache, bukan penyimpanan permanen.
- **Status video/klip berbentuk state machine linear** (`UPLOADED -> TRANSCRIBED -> CLIPS_DETECTED -> RENDERED`) yang disimpan di PostgreSQL agar frontend bisa polling progres secara konsisten.
- **Prisma di `packages/database` sebagai satu-satunya akses ke PostgreSQL**, dipakai baik oleh `apps/api` maupun `apps/worker` (model: `User`, `Video`, `TranscriptSegment`, `Clip`, `SocialAccount` â€” lihat `packages/database/prisma/schema.prisma`). Transcript segment disimpan per-video (bukan diduplikasi per-klip); transcript sebuah klip didapat dengan query segment dalam rentang `startTime`-`endTime` klip tersebut.
- **Video disimpan di object storage S3-compatible** (`packages/storage`, dipakai bareng oleh `apps/api` dan `apps/worker`), bukan local disk â€” supaya `apps/api` dan `apps/worker` (proses terpisah, bisa jalan di mesin/container berbeda) tidak perlu berbagi filesystem. `apps/api/src/storage` upload buffer hasil multer langsung ke bucket lewat `uploadObject()`, `Video.sourceUrl` menyimpan **object key** (`videos/<uuid>.<ext>`), bukan path. Konfigurasi lewat env var generik `STORAGE_*` (`STORAGE_ENDPOINT`, `STORAGE_REGION`, `STORAGE_BUCKET`, `STORAGE_ACCESS_KEY_ID`, `STORAGE_SECRET_ACCESS_KEY`, `STORAGE_FORCE_PATH_STYLE`) â€” bukan `R2_*` â€” supaya provider-nya bisa diganti (mis. AWS S3) tanpa ubah kode, walau dev saat ini dites terhadap Cloudflare R2 asli. `packages/storage` pakai lazy singleton (`getClient()`, konstruksi `S3Client` ditunda sampai first use) untuk hindari bug class baca `process.env` sebelum `.env` ke-load â€” sama seperti kasus `QueueModule`/`JwtStrategy`.
- **`apps/worker` baca source video langsung dari object storage tanpa nyentuh disk untuk job `transcribe`** â€” `getObjectStream()` mengembalikan Node `Readable`, di-pipe langsung ke `toFile()` (helper dari OpenAI SDK yang terima `AsyncIterable`) lalu masuk ke Whisper API. Untuk job `render-clip`, FFmpeg **butuh path file lokal asli** (tidak bisa seek/tulis langsung ke object storage), jadi `apps/worker/src/storage.ts` cuma menyediakan **scratch space sementara** di `os.tmpdir()` (`reserveScratchPath()`) â€” source didownload ke situ, FFmpeg proses di situ, hasil render diupload balik ke bucket (`renders/<clipId>.mp4`), lalu semua file scratch (source/SRT/output) dihapus di blok `finally` terlepas sukses/gagal.
- **Worker meng-update status video sendiri** setelah job selesai (mis. `transcribe` job set status `TRANSCRIBED` setelah berhasil, atau `FAILED` kalau error), bukan lewat callback ke `apps/api`.
- **Worker self-chains job berikutnya dalam pipeline**: `transcribe` job yang sukses langsung enqueue job `detect-clips` sendiri, dan `detect-clips` yang sukses langsung enqueue satu job `render-clip` per kandidat klip (lihat `apps/worker/src/queues.ts`), bukan lewat `apps/api`. Orkestrasi antar-tahap pipeline berada di `apps/worker`, bukan di layer API.
- **FFmpeg dipanggil langsung via `child_process`** (bukan wrapper library seperti `fluent-ffmpeg`) dari `apps/worker/src/ffmpeg.ts`. Butuh binary FFmpeg tersedia (di `PATH`, atau via `FFMPEG_PATH`) di environment tempat `apps/worker` jalan.
- **Render output diupload ke object storage juga** (prefix `renders/`, key `renders/<clipId>.mp4`), konsisten dengan keputusan storage video asli â€” `apps/worker` satu-satunya penulis object ini.
- **`GET /videos/:id` tidak mengembalikan `Clip.outputUrl` mentah** (object storage key) ke client; di-map jadi `downloadUrl` (`/clips/:id/download` relatif) supaya detail implementasi storage tidak bocor ke API response. `GET /clips/:id/download` di `apps/api` stream file dari bucket lewat `getObjectStream()` (bukan `res.download()` ke path lokal lagi).
- **CORS di `apps/api` di-enable eksplisit dengan `credentials: true`** (`WEB_ORIGIN`, default `http://localhost:3000`) supaya `apps/web` (origin beda, port 3000) bisa manggil `apps/api` (port 3001) dari browser sambil ikut kirim cookie sesi.
- **Auth: email + password + JWT di httpOnly cookie**, bukan token di `localStorage`/`Authorization` header. `AuthModule` (`apps/api/src/auth`) â€” `POST /auth/register` (bcrypt hash, langsung login), `POST /auth/login`, `POST /auth/logout`, `GET /auth/me`. `JwtStrategy` extract token dari cookie `token`, bukan `Bearer` header. `ownerId` untuk video **selalu** diambil dari user di sesi (`@CurrentUser()`), tidak pernah dari body request â€” endpoint lama `POST /users` (get-or-create by email tanpa password) sudah dihapus karena itu lubang keamanan (siapa saja bisa "jadi" user manapun cuma dengan tahu emailnya).
- **`GET /videos/:id` dan `GET /clips/:id/download` mengecek kepemilikan** (`video.ownerId`/`clip.video.ownerId` harus sama dengan user di sesi) dan melempar 404 yang sama baik untuk resource yang tidak ada maupun resource milik user lain â€” supaya endpoint tidak bisa dipakai untuk menebak ID mana yang valid.
- **`apps/web` punya dua route**: `/` (upload video baru + progress live untuk video yang baru saja di-upload) dan `/dashboard` (riwayat semua video milik user, lewat `GET /videos`). Keduanya pakai `useAuth()` hook yang sama (`lib/useAuth.ts`) untuk cek sesi via `GET /auth/me` â€” bukan `localStorage` â€” supaya status login selalu konsisten dengan sesi di server.
- **`POST /auth/login` di-rate-limit** (5 percobaan / 60 detik, per IP, via `@nestjs/throttler`). `ThrottlerGuard` **tidak** didaftarkan global â€” cuma dipasang `@UseGuards(ThrottlerGuard)` langsung di route `login`, jadi endpoint lain (termasuk `/auth/register`) tidak kena efeknya. Config in-memory (bukan Redis-backed) karena `apps/api` masih jalan sebagai satu instance untuk MVP ini.
- **Env var wajib divalidasi saat boot, gagal cepat kalau ada yang kosong** â€” `apps/api/src/config/env.validation.ts` (dipasang via `ConfigModule.forRoot({ validate })`, jalan sinkron sebelum modul lain di-build) dan `apps/worker/src/env.ts` (dipanggil di `main.ts` tepat setelah `dotenv.config()`, sebelum modul lain yang baca `process.env` di-import). Cuma var tanpa fallback aman yang wajib (`DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`/`OPENAI_API_KEY`, `STORAGE_*`) â€” var yang sudah punya default masuk akal di kode (`WEB_ORIGIN`, `API_PORT`, `JWT_EXPIRES_IN`, `FFMPEG_PATH`) tidak diwajibkan. Tujuannya supaya `JWT_SECRET` kosong (yang tanpa ini akan diam-diam sign token dengan `undefined`) atau `STORAGE_*` kosong (yang tanpa ini akan gagal jauh di dalam percobaan koneksi S3) langsung ketahuan di boot, bukan nanti pas runtime.
- **`apps/api` pakai `helmet()`** (security response headers: CSP, HSTS, X-Frame-Options, dst â€” lihat `apps/api/src/main.ts`) dan punya endpoint **`GET /health`** (tidak perlu auth, tidak di-rate-limit â€” dipanggil load balancer/orchestrator) yang query `SELECT 1` ke Postgres dan balas `503` kalau DB tidak bisa dijangkau.
- **Tiap app punya `Dockerfile` multi-stage sendiri**, di-build dari root repo (workspace deps butuh akses ke `packages/*`). Pakai `pnpm deploy --prod --ignore-scripts` untuk bundel dist + production deps app tersebut jadi direktori yang self-contained tanpa symlink workspace (`--ignore-scripts` karena kalau tidak, postinstall `packages/database` â€” `prisma generate` â€” jalan ulang di install `--prod`, padahal `prisma` CLI-nya sendiri devDependency yang sudah kepangkas; generated client dari build step sebelumnya sudah ikut ter-copy apa adanya). Tidak ada `.env` yang di-copy ke image manapun â€” `envFilePath`/`dotenv.config()` yang dipakai `apps/api`/`apps/worker` cuma no-op kalau file-nya tidak ada, sama seperti perilakunya di CI, dan config sepenuhnya datang dari environment variable asli yang dikasih saat container jalan. `apps/worker`'s image nginstall `ffmpeg` asli lewat `apk` â€” kalau override `FFMPEG_PATH`, pastikan itu path di *dalam* container, bukan path host (gampang kebawa kalau reuse `.env` dev apa adanya buat container). `apps/web`'s image pakai Next.js `output: 'standalone'` (lihat `next.config.mjs`) plus `NEXT_PUBLIC_API_URL` sebagai build arg (bukan runtime env â€” `NEXT_PUBLIC_*` di-inline ke bundle client saat `next build`). Migration jalan lewat `packages/database/Dockerfile` terpisah (single-stage, full install biar `prisma` CLI-nya ada) sebagai one-shot job sebelum `apps/api`/`apps/worker` start â€” lihat `docker-compose.prod.yml`, service `migrate` pakai `condition: service_completed_successfully` supaya api/worker nunggu migration selesai dulu.

## Konvensi Coding

- Bahasa: TypeScript di seluruh monorepo (`apps/web`, `apps/api`, `apps/worker`, `packages/shared`).
- Semua kontrak data (job payload, DTO API, enum status) didefinisikan di `packages/shared` dan diimpor, bukan diduplikasi.
- Job BullMQ dinamai dengan verb-noun (`transcribe`, `detect-clips`, `render-clip`) dan payload/return type-nya didefinisikan di `packages/shared`.
- Perubahan skema PostgreSQL melalui migration (bukan sync otomatis) agar histori skema terlacak.
